# 5월 28일 필기 — JSP / Servlet

## 핵심요약

03_FILTER에서는 필터(Filter)를 학습했다. 필터는 `javax.servlet.Filter` 인터페이스를 구현하며, `chain.doFilter()` 호출 전이 요청 전처리, 호출 후가 응답 후처리 구간이다. 여러 필터를 같은 URL에 등록하면 `web.xml` 등록 순서대로 체인(Chain) 방식으로 순차 실행되고, 응답 시에는 역순으로 실행된다. `@WebFilter`는 단일 필터에 간편하지만 다중 필터의 순서 보장이 안 되므로 순서가 중요할 때는 `web.xml` 등록 방식을 써야 한다. `UTF8_EncodingFilter`는 모든 `.do` 요청에 `request.setCharacterEncoding("UTF-8")`을 일괄 적용해 서블릿마다 인코딩 코드를 반복하는 것을 막는 대표적인 필터 활용 패턴이다. `url-pattern`에 `*.do`가 아닌 `.do`를 쓰면 `IllegalArgumentException`이 발생해 앱 전체가 배포 실패한다. 04_LISTENER에서는 서블릿 컨테이너의 생명주기 이벤트를 감지하는 리스너 6종을 학습했다. 앱 시작/종료를 감지하는 `ServletContextListener`, 세션 생성/소멸을 감지하는 `HttpSessionListener`, 요청 시작/종료를 감지하는 `ServletRequestListener`가 핵심이며, 각각에 대응하는 Attribute 리스너 3개도 존재한다. `@WebListener` 어노테이션으로 등록하며, 현재 프로젝트 C01~C06은 골격 코드가 주석 처리된 상태다. 05_RESOURCE에서는 JNDI를 통한 커넥션 풀(Connection Pool)을 학습했다. 직접 JDBC는 요청마다 TCP 커넥션을 새로 생성해 느리지만, DataSource는 Tomcat이 앱 시작 시 커넥션을 미리 만들어 풀로 관리하므로 빠르다. 설정은 `META-INF/context.xml`의 `<Resource>`에 `maxActive`(최대 활성), `maxIdle`(최대 유휴), `maxWait`(대기 시간) 속성으로 지정하고, `InitialContext`로 `java:/comp/env`에 진입한 뒤 `jdbc/MysqlDB` 키로 `DataSource`를 조회해 커넥션을 얻는다. `conn.close()`는 실제 연결 종료가 아니라 풀에 반납하는 것이다. `DbUtils`의 `conn`, `pstmt`, `rs`를 `static` 필드로 선언하면 동시 요청 시 덮어씌워지는 스레드 안전 문제가 생기므로 실무에서는 메서드 지역변수 또는 `try-with-resources` 패턴을 써야 한다.

## JSP02 — 03_FILTER 필터 (Filter)

---

### 1) 핵심 개념

| 개념 | 내용 |
|---|---|
| **Filter** | 서블릿 요청/응답 사이에서 전처리·후처리를 수행하는 컴포넌트 |
| **구현 방법** | `javax.servlet.Filter` 인터페이스 구현 → `doFilter()` 오버라이드 |
| **chain.doFilter()** | 다음 필터 또는 서블릿으로 요청을 넘기는 메서드. 호출 전 = 전처리, 후 = 후처리 |
| **실행 순서** | `web.xml` 등록 순서대로 실행, 응답은 역순 |

---

### 2) 필터 실행 구조

```
클라이언트 요청
    │
    ▼
[ Filter_1 전처리 ]  ─ chain.doFilter() 호출 전
    │
    ▼
[ Filter_2 전처리 ]  ─ chain.doFilter() 호출 전
    │
    ▼
  Servlet (비즈니스 로직)
    │
    ▼
[ Filter_2 후처리 ]  ─ chain.doFilter() 호출 후
    │
    ▼
[ Filter_1 후처리 ]  ─ chain.doFilter() 호출 후
    │
    ▼
클라이언트 응답
```

`/main.do`에 Filter_1, Filter_2를 등록했을 때 콘솔 출력 순서:

```
[FILTER] MAIN FILTER_1 START
[FILTER] MAIN FILTER_2 START
  → 서블릿 실행
[FILTER] MAIN FILTER_2 END
[FILTER] MAIN FILTER_1 END
```

---

### 3) MainFilter_1 — 기본 필터 구조

```java
public class MainFilter_1 implements Filter {

    @Override
    public void doFilter(ServletRequest request, ServletResponse response, FilterChain chain)
            throws IOException, ServletException {

        // ① 요청 전처리
        System.out.println("[FILTER] MAIN FILTER_1 START");

        chain.doFilter(request, response);   // ② 다음 필터 or 서블릿으로 전달

        // ③ 응답 후처리
        System.out.println("[FILTER] MAIN FILTER_1 END");
    }
}
```

---

### 4) UTF8_EncodingFilter — 인코딩 공통 처리

```java
public class UTF8_EncodingFilter implements Filter {

    @Override
    public void doFilter(ServletRequest request, ServletResponse response, FilterChain chain)
            throws IOException, ServletException {

        // 요청 전 — 한글 파라미터 깨짐 방지
        request.setCharacterEncoding("UTF-8");

        chain.doFilter(request, response);

        // 응답 후 — 응답 Content-Type 설정
        response.setContentType("text/html; charset=UTF-8");
    }
}
```

> 모든 서블릿마다 `request.setCharacterEncoding("UTF-8")`을 반복하는 대신  
> 필터 1개에 `*.do` 패턴을 적용해 전체 서블릿에 일괄 처리

---

### 5) 필터 등록 방법 비교

#### 방법 1 — `@WebFilter` 어노테이션

```java
@WebFilter("/main.do")
public class MainFilter_1 implements Filter { ... }

// 여러 URL 동시 적용
@WebFilter(urlPatterns = {"/main.do", "/login.do"})
public class MainFilter_1 implements Filter { ... }
```

#### 방법 2 — `web.xml` 등록

```xml
<filter>
    <filter-name>UTF8_EncodingFilter</filter-name>
    <filter-class>Filter.UTF8_EncodingFilter</filter-class>
</filter>
<filter-mapping>
    <filter-name>UTF8_EncodingFilter</filter-name>
    <url-pattern>*.do</url-pattern>
</filter-mapping>
```

| 구분 | `@WebFilter` | `web.xml` |
|---|---|---|
| **다중 필터 순서** | 보장 안 됨 (JVM 로딩 순서 의존) | 등록 순서대로 보장 |
| **URL 변경 시** | 코드 재컴파일 필요 | 설정 파일만 수정 |
| **권장 상황** | 단순 단일 필터 | 순서가 중요한 다중 필터 |

---

### 6) ⚠ url-pattern 주의사항

| 패턴 | 적용 범위 |
|---|---|
| `/main.do` | 해당 URL만 정확히 일치 |
| `*.do` | `.do`로 끝나는 모든 URL |
| `/*` | 모든 URL |
| `.do` | **잘못된 형식** — `IllegalArgumentException` 발생, 앱 전체 배포 실패 |

> Servlet `url-pattern`과 동일한 규칙 — `*.` 확장자 패턴은 반드시 `*`로 시작해야 함

## JSP02 — 04_LISTENER 리스너 (Listener)

---

### 1) 핵심 개념

| 개념 | 내용 |
|---|---|
| **Listener** | 서블릿 컨테이너의 생명주기 이벤트를 감지해 자동 호출되는 컴포넌트 |
| **등록 방법** | `@WebListener` 어노테이션 또는 `web.xml` |
| **용도** | 앱 시작 시 초기화, 동시 접속자 수 카운트, 세션 만료 처리 등 |

---

### 2) 리스너 종류 6가지

| 클래스 | 인터페이스 | 감지 이벤트 | 주요 메서드 |
|---|---|---|---|
| C01 | `ServletContextListener` | 앱 시작 / 종료 | `contextInitialized()`, `contextDestroyed()` |
| C02 | `ServletContextAttributeListener` | application 속성 변경 | `attributeAdded()`, `attributeRemoved()`, `attributeReplaced()` |
| C03 | `HttpSessionListener` | 세션 생성 / 소멸 | `sessionCreated()`, `sessionDestroyed()` |
| C04 | `HttpSessionAttributeListener` | session 속성 변경 | `attributeAdded()`, `attributeRemoved()`, `attributeReplaced()` |
| C05 | `ServletRequestListener` | 요청 시작 / 종료 | `requestInitialized()`, `requestDestroyed()` |
| C06 | `ServletRequestAttributeListener` | request 속성 변경 | `attributeAdded()`, `attributeRemoved()`, `attributeReplaced()` |

---

### 3) 이벤트 발생 범위 (생명주기 계층)

```
[ 앱 시작 ]  →  C01 contextInitialized()
    │
    │  [ 요청 수신 ]  →  C05 requestInitialized()
    │      │
    │      │  [ 세션 생성 ]  →  C03 sessionCreated()
    │      │      │
    │      │      │  session.setAttribute()  →  C04 attributeAdded()
    │      │      │
    │      │  [ 세션 소멸 ]  →  C03 sessionDestroyed()
    │      │
    │  [ 요청 완료 ]  →  C05 requestDestroyed()
    │
[ 앱 종료 ]  →  C01 contextDestroyed()
```

---

### 4) 등록 방법 및 코드 예시

```java
// C01 — 앱 시작/종료 감지 (서버 구동 시 초기화 작업에 활용)
@WebListener
public class C01ServletContextListener implements ServletContextListener {

    @Override
    public void contextInitialized(ServletContextEvent sce) {
        // 앱 시작 시 1회 — DB 커넥션 풀 초기화, 공통 설정 로딩 등
        System.out.println("[LISTENER] C01ServletContextListener..start...");
    }

    @Override
    public void contextDestroyed(ServletContextEvent sce) {
        // 앱 종료 시 1회 — 자원 해제
        System.out.println("[LISTENER] C01ServletContextListener..end...");
    }
}
```

```java
// C03 — 세션 생성/소멸 감지 (동시 접속자 수 카운트 등에 활용)
@WebListener
public class C03HttpSessionListener implements HttpSessionListener {

    @Override
    public void sessionCreated(HttpSessionEvent se) {
        System.out.println("[LISTENER] C03HttpSessionListener Created...");
    }

    @Override
    public void sessionDestroyed(HttpSessionEvent se) {
        System.out.println("[LISTENER] C03HttpSessionListener Destroyed...");
    }
}
```

```java
// C02 — application 속성 변경 감지
@WebListener
public class C02ServletContextAttributeListener implements ServletContextAttributeListener {

    @Override
    public void attributeAdded(ServletContextAttributeEvent scae) {
        System.out.println("[LISTENER] C02 add()..");
    }

    @Override
    public void attributeRemoved(ServletContextAttributeEvent scae) {
        System.out.println("[LISTENER] C02 remove()..");
    }

    @Override
    public void attributeReplaced(ServletContextAttributeEvent scae) {
        System.out.println("[LISTENER] C02 replace()..");
    }
}
```

> 현재 프로젝트의 C01~C06 리스너는 모두 주석 처리된 학습용 골격 코드 상태

## JSP02 — 05_RESOURCE 커넥션 풀 / DataSource (JNDI)

---

### 1) 핵심 개념

| 개념 | 내용 |
|---|---|
| **커넥션 풀** | Tomcat이 앱 시작 시 DB 커넥션을 미리 여러 개 만들어 풀로 관리 |
| **DataSource** | 커넥션 풀을 표준 인터페이스(`javax.sql.DataSource`)로 추상화한 것 |
| **JNDI** | Java Naming and Directory Interface — 설정된 자원을 이름으로 조회하는 표준 메커니즘 |
| **설정 파일** | `META-INF/context.xml` |

---

### 2) 직접 JDBC vs 커넥션 풀 비교

| 구분 | 직접 JDBC | 커넥션 풀 (DataSource) |
|---|---|---|
| **커넥션 생성 시점** | 요청마다 새로 생성 | 앱 시작 시 미리 생성 |
| **속도** | 느림 (TCP 핸드셰이크 매번) | 빠름 (풀에서 재사용) |
| **설정 위치** | Java 코드 (`Class.forName`, `DriverManager`) | `context.xml` |
| **관리 주체** | 개발자 직접 close | Tomcat 컨테이너 |

---

### 3) META-INF/context.xml — 커넥션 풀 설정

```xml
<Context>
    <Resource
        driverClassName="com.mysql.cj.jdbc.Driver"
        url="jdbc:mysql://localhost/opendatadb"
        username="root"
        password="1234"
        name="jdbc/MysqlDB"
        type="javax.sql.DataSource"
        auth="Container"
        maxActive="10"
        maxIdle="2"
        maxWait="5000" />
</Context>
```

| 속성 | 설명 |
|---|---|
| `name` | JNDI 조회 키 — Java 코드에서 `lookup("jdbc/MysqlDB")`로 찾음 |
| `type` | `javax.sql.DataSource` 고정 |
| `auth` | `Container` — Tomcat이 인증 관리 |
| `maxActive` | 풀에서 동시에 사용 가능한 최대 커넥션 수 |
| `maxIdle` | 풀에서 대기 상태로 유지할 최대 커넥션 수 |
| `maxWait` | 풀 고갈 시 커넥션을 기다리는 최대 시간 (ms), `-1`이면 무한 대기 |

---

### 4) DbUtils.conn() — JNDI 조회

```java
public static void conn() throws Exception {
    // ① JNDI 네이밍 컨텍스트 진입
    Context initContext = new InitialContext();
    Context envContext  = (Context) initContext.lookup("java:/comp/env");

    // ② DataSource 조회 (context.xml의 name 속성값과 일치해야 함)
    DataSource dataSource = (DataSource) envContext.lookup("jdbc/MysqlDB");

    // ③ 풀에서 커넥션 1개 대여
    conn = dataSource.getConnection();
    System.out.println("Conn : " + conn);
}
```

| JNDI 경로 | 설명 |
|---|---|
| `java:/comp/env` | J2EE 표준 환경 네이밍 컨텍스트 진입점 (고정값) |
| `jdbc/MysqlDB` | context.xml `name` 속성값과 반드시 일치해야 함 |

> `dataSource.getConnection()` = 풀에서 커넥션을 빌리는 것  
> `conn.close()` = 실제 연결 종료가 아닌 **풀에 반납**하는 것 — 반드시 호출해야 함

---

### 5) DbUtils 전체 구조

```java
public class DbUtils {
    private static Connection conn = null;
    private static PreparedStatement pstmt = null;
    private static ResultSet rs = null;

    // ① 커넥션 획득 (JNDI)
    public static void conn() throws Exception { ... }

    // ② 자원 일괄 해제 (풀 반납)
    public static void disConn() throws Exception {
        if (rs    != null) rs.close();
        if (pstmt != null) pstmt.close();
        if (conn  != null) conn.close();
    }

    // ③ INSERT
    public static int insertUser(String username, String password) throws Exception {
        pstmt = conn.prepareStatement("insert into tbl_user values(?,?,?,?,?,?)");
        pstmt.setString(1, username);
        pstmt.setString(2, password);
        // ...
        return pstmt.executeUpdate();
    }

    // ④ SELECT
    public static UserDto selectUser(String username) throws Exception {
        pstmt = conn.prepareStatement("select * from tbl_user where userid=?");
        pstmt.setString(1, username);
        rs = pstmt.executeQuery();
        UserDto dto = null;
        if (rs != null && rs.next()) {
            dto = new UserDto();
            dto.setUsername(rs.getString("username"));
            dto.setPassword(rs.getString("password"));
        }
        return dto;
    }
}
```

### 6) JSP에서 DataSource 직접 사용

```jsp
<%@ page import="Servlet.*" %>
<%
    DbUtils.conn();        // JNDI로 풀에서 커넥션 획득
    // ... DB 작업 (insert, select 등)
    DbUtils.disConn();     // 자원 반납
%>
```

### 7) ⚠ 설계 주의사항 — static 필드 스레드 안전 문제

```java
// 현재 DbUtils 구조 (문제 있음)
private static Connection conn = null;       // static 필드
private static PreparedStatement pstmt = null;
private static ResultSet rs = null;
```

`static` 필드는 클래스 공유 → 동시 요청 시 다른 요청의 값으로 **덮어씌워질 수 있음**

```java
// 권장 패턴 — 메서드 지역변수 + try-with-resources
public static Connection getConn() throws Exception {
    Context envContext = (Context) new InitialContext().lookup("java:/comp/env");
    return ((DataSource) envContext.lookup("jdbc/MysqlDB")).getConnection();
}

// 사용 예
try (Connection conn = DbUtils.getConn();
     PreparedStatement pstmt = conn.prepareStatement("SELECT ...")) {
    // ... 사용
}   // try 블록 종료 시 자동으로 conn.close() → 풀 반납
```
