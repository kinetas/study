# 5월 6일 필기 — JAVA

## 핵심요약

추상 클래스는 `abstract` 키워드로 선언하며 직접 객체 생성이 불가하고 하위 클래스가 반드시 추상 메서드를 재정의해야 한다. 인터페이스는 모든 메서드가 자동으로 `public abstract`, 필드는 `public static final`이 적용되며 클래스와 달리 `implements`로 다중 구현이 가능하다. `extends`와 `implements`를 함께 쓸 때는 `extends`가 반드시 먼저 위치해야 하고, 인터페이스 타입으로 필드를 선언하면 구현 클래스를 런타임에 자유롭게 교체할 수 있다. `final` 키워드는 적용 대상에 따라 변수는 값 변경 불가, 메서드는 오버라이딩 불가, 클래스는 상속 불가를 의미한다. 인터페이스를 먼저 설계하면 구현이 완성되기 전에도 팀원이 각자 다른 파트를 독립적으로 작업할 수 있으며, 하나의 클래스가 여러 인터페이스를 동시에 구현하면 같은 객체를 어느 타입으로든 캐스팅해 참조할 수 있다.

### 1) 추상 클래스 (Abstract Class)

일반 클래스와 달리 **직접 객체 생성 불가** — 하위 클래스가 추상 메서드를 재정의해야만 인스턴스화 가능.

| 구분 | 일반 클래스 | 추상 클래스 |
|---|---|---|
| 직접 객체 생성 | O | X |
| 추상 메서드 | 없음 | 있음 (하위 클래스 강제 구현) |
| UpCasting | O | O |
| 용도 | 일반 상속 | 설계 강제 / 공통 뼈대 제공 |

```java
// 일반 클래스 — 상위 클래스도 new 가능
class Parent1 { void func() {} }
class Son1 extends Parent1 { void func() { System.out.println("Son1"); } }

// 추상 클래스 — 상위 클래스는 new 불가
abstract class Parent2 { abstract void func(); }
class Son2 extends Parent2 { @Override void func() { System.out.println("Son2"); } }
class Son3 extends Parent2 { @Override void func() { System.out.println("Son3"); } }
```

```java
Parent1 ob3 = new Son1();   // UpCasting O
ob3.func();                 // "Son1" (동적 바인딩)

// Parent2 ob4 = new Parent2(); // 컴파일 에러 — 추상 클래스 new 불가
Parent2 ob6 = new Son2();   // UpCasting O
ob6.func();                 // "Son2"
ob6 = new Son3();           // 같은 참조변수에 다른 하위 객체 대입
ob6.func();                 // "Son3"
```

### 2) 추상 클래스 활용 — 템플릿 메서드 패턴 (C02)

변환 로직(`convert`)과 UI 흐름(`run`)을 분리 — 하위 클래스는 변환 공식만 구현하면 된다.

```java
abstract class Converter {
    abstract protected double convert(double src);
    abstract protected String getSrcString();
    abstract protected String getDestString();
    protected double ratio;

    public void run() {
        Scanner scanner = new Scanner(System.in);
        System.out.print(getSrcString() + "을 입력하세요>> ");
        double val = scanner.nextDouble();
        System.out.println("변환 결과: " + convert(val) + getDestString());
        scanner.close();
    }
}
```

```java
// 원화 → 달러 : ratio를 생성자로 주입
class Won2Dollar extends Converter {
    public Won2Dollar(int ratio) { this.ratio = ratio; }
    protected double convert(double src) { return src / ratio; }
    protected String getSrcString()      { return "원"; }
    protected String getDestString()     { return "달러"; }
}

// 섭씨 → 화씨 : ratio 불필요, 고정 공식 사용
class 섭씨to화씨 extends Converter {
    public 섭씨to화씨() { super(); }  // 기본 생성자 명시적 호출
    protected double convert(double src) { return (src * 1.8) + 32; }
    protected String getSrcString()      { return "섭씨"; }
    protected String getDestString()     { return "화씨"; }
}
```

> `Won2Dollar`는 ratio를 외부에서 주입받아 유연하게 환율 변경 가능,
> `섭씨to화씨`는 ratio 미사용 — 공식이 고정이므로 `super()`만 호출

### 3) 인터페이스 (Interface)

클래스는 단일 상속만 가능하지만, 인터페이스는 **다중 구현(implements)** 가능.

```java
interface Remocon {
    void powerOn();             // 자동으로 public abstract
    void powerOff();
    void setVolumn(int vol);
    int MAX_VOL = 100;          // 자동으로 public static final
    int MIN_VOL = 0;
}
interface Browser { void searchURL(String url); }
```

```java
class Tv implements Remocon {
    private int volumn;
    @Override public void powerOn()  { System.out.println("TV on"); }
    @Override public void powerOff() { System.out.println("TV off"); }
    @Override public void setVolumn(int vol) {
        if      (MAX_VOL < vol) this.volumn = MAX_VOL;  // 인터페이스 상수 바로 사용
        else if (MIN_VOL > vol) this.volumn = MIN_VOL;
        else                    this.volumn = vol;
        System.out.println("TV 현재 볼륨 : " + this.volumn);
    }
}

// extends 가 implements 보다 반드시 앞에 위치
class SmartTv extends Tv implements Browser {
    @Override public void searchURL(String url) { System.out.println(url + "로 이동합니다."); }
}
```

```java
// 인터페이스 타입 파라미터로 다형성 활용
public static void TurnOn(Remocon remocon)               { remocon.powerOn(); }
public static void ChangeVolumn(Remocon remocon, int vol) { remocon.setVolumn(vol); }
public static void WebBrowser(Browser browser, String url){ browser.searchURL(url); }

SmartTv smartTv = new SmartTv();
TurnOn(smartTv);                     // SmartTv → Remocon 으로 UpCasting
ChangeVolumn(smartTv, -5);           // -5 → MIN_VOL(0) 으로 보정
WebBrowser(smartTv, "https://www.naver.com");
```

### 4) 인터페이스 다형성 — 부품 교체 패턴 (C04)

인터페이스 타입으로 필드를 선언하면 구현 클래스를 런타임에 자유롭게 교체 가능.

```java
interface Tire { void run(); }
class 한국타이어 implements Tire { public void run() { System.out.println("한국타이어가 굴러갑니다"); } }
class 금호타이어 implements Tire { public void run() { System.out.println("금호타이어가 굴러갑니다"); } }

class Car {
    Tire FL, FR, BL, BR;    // 인터페이스 타입 — 어떤 Tire 구현체든 대입 가능
    Car() { FL = FR = BL = BR = new 한국타이어(); }
    void start() { FL.run(); FR.run(); BL.run(); BR.run(); }
}
```

```java
Car car = new Car();
car.start();               // 전체 한국타이어
car.FR = new 금호타이어(); // 특정 바퀴만 교체 — Car 코드 수정 없이 가능
car.BL = new 금호타이어();
car.start();               // FL·BR = 한국, FR·BL = 금호
```

> Car는 Tire 인터페이스에만 의존 → 새 타이어 브랜드 추가 시 Car 수정 불필요 (OCP 원칙)

### 5) 클래스 + 인터페이스 다중 상속 규칙 (C05)

```java
// extends 하나만, implements 는 여러 개 가능
class E extends C implements A, B {}

// class E extends C, D implements A, B {}  // 컴파일 에러 — 클래스 다중 상속 불가
```

| 구분 | 클래스(extends) | 인터페이스(implements) |
|---|---|---|
| 다중 가능 | X (하나만) | O (여러 개) |
| 순서 | 먼저 | 나중 |

### 6) final 키워드 (C07)

`final`은 적용 대상에 따라 의미가 다르다.

```java
class Parent {
    int n1 = 100;
    final int n2 = 200;     // 심볼릭 상수 — 재할당 불가

    void setN2(int n2) {
        // this.n2 = n2;    // 컴파일 에러
    }

    final void func1() {    // 하위 클래스에서 재정의 불가
        System.out.println("Parent's final func1");
    }
}

final class Son extends Parent {
    // final void func1() {} // 컴파일 에러 — 상위의 final 메서드 재정의 불가
}
// class EndPoint extends Son {} // 컴파일 에러 — final 클래스 상속 불가
```

| 적용 대상 | 의미 |
|---|---|
| `final` 변수 | 값 변경 불가 (상수) |
| `final` 메서드 | 하위 클래스 오버라이딩 불가 |
| `final` 클래스 | 상속 불가 (예: `String`, `Integer`) |

### 1) 실습 의도 — 파트 분담 구조

인터페이스를 먼저 설계하면 구현이 끝나지 않아도 팀원이 각자 다른 파트를 독립적으로 작업할 수 있다.

```
팀장
 ├── Common/Information.java       공통 메타정보 인터페이스 설계
 ├── Interfaces/SumInterface.java  덧셈 담당자에게 줄 계약서
 ├── Interfaces/SubInterface.java  뺄셈 담당자에게 줄 계약서
 ├── Interfaces/MulInterface.java  곱셈 담당자에게 줄 계약서
 ├── Interfaces/DivInterface.java  나눗셈 담당자에게 줄 계약서
 └── Main.java                     완성된 구현체를 모아서 테스트

팀원 A → SumImpl 구현
팀원 B → SubImpl 구현
팀원 C → MulImpl 구현
팀원 D → DivImpl 구현
```

> 팀원은 자기 인터페이스 파일만 받으면 된다.
> Main도 인터페이스 타입으로만 선언해두면 구현체가 나중에 붙어도 코드 변경 없음.

---

### 2) 팀장 — 공통 인터페이스 설계

```java
// Common/Information.java — Main에서 누구 구현인지 식별 가능하게
public interface Information {
    String getTeamName();
    String getUsername();
}
```

```java
// Interfaces/SumInterface.java — 오버로딩으로 4가지 시그니처 정의
public interface SumInterface {
    void sum(int a, int b);           // 2개 int
    void sum(int a, int b, int c);    // 3개 int
    void sum(int... arg);             // 가변인자 int
    void sum(String... arg);          // 가변인자 String (숫자 문자열)
}
// SubInterface / MulInterface / DivInterface 도 동일한 구조
```

> 같은 이름 `sum`에 파라미터만 다른 것 → **메서드 오버로딩** (컴파일 타임 구분)

---

### 3) 팀원 — 자기 파트 구현

`SumInterface`와 `Information`을 **동시에 implements** — Main에서 이름 조회가 가능하려면 필수.

```java
public class SumImpl implements SumInterface, Information {

    @Override
    public void sum(int a, int b) { System.out.println("a + b = " + (a + b)); }

    @Override
    public void sum(int a, int b, int c) { System.out.println("a + b + c = " + (a + b + c)); }

    @Override
    public void sum(int... arg) {
        int s = 0;
        for (int numb : arg) s += numb;   // enhanced for
        System.out.println("총 합 : " + s);
    }

    @Override
    public void sum(String... arg) {
        // String s = "";
        // for (String sentence : arg) s += sentence; // 이어붙이기 방식 먼저 떠올렸다가
        int s = 0;
        for (String numb : arg) s += Integer.parseInt(numb); // 숫자 합산이 의도 → parseInt 로 변경
        System.out.println(s);
    }

    @Override public String getTeamName() { return "2조"; }
    @Override public String getUsername()  { return "김정희"; }
}
```

### 4) 팀장 — Main에서 구현체 조립 및 테스트

```java
public class Main {
    static SumInterface 김정희_sum;
    // static SubInterface 팀원B_sub;  ← 각자 담당 파트 타입으로 선언
    // static MulInterface 팀원C_mul;
    // static DivInterface 팀원D_div;

    static void init() {
        김정희_sum = (SumInterface) new Ch08.Team0.Impl.김정희.SumImpl();
    }

    static void test_sum(SumInterface sumImpl) {
        // SumImpl은 SumInterface이면서 동시에 Information — 같은 객체를 캐스팅해 이름 조회
        Information info = (Information) sumImpl;
        System.out.printf("======= %s / %s ========\n", info.getTeamName(), info.getUsername());
        sumImpl.sum(10, 20);
        sumImpl.sum(10, 20, 30, 40);
        sumImpl.sum(10, 20, 30, 40, 50);
        sumImpl.sum("11", "22", "33", "44");
    }

    public static void main(String[] args) {
        init();
        SumInterface[] impls = { 김정희_sum /*, 다른팀원_sum ... */ };
        for (SumInterface impl : impls) test_sum(impl);
    }
}
```

**`(Information) sumImpl` 캐스팅이 가능한 이유**

```
SumImpl 객체
  └── implements SumInterface  →  SumInterface 타입으로 참조 가능
  └── implements Information   →  Information 타입으로도 캐스팅 가능
```

하나의 객체가 두 인터페이스를 모두 구현 → 어느 타입으로든 참조 가능.

---

> **추상 클래스 vs 인터페이스 선택 기준**
> - 공통 필드나 구현 코드를 공유해야 할 때 → **추상 클래스**
> - 타입만 통일하고 다중 구현이 필요할 때 → **인터페이스**
