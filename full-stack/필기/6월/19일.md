# 6월 19일 필기 — Spring Boot / Listener · HandlerMapping

## 핵심요약

오늘은 Spring Boot의 Listener(17)와 HandlerMapping 커스터마이징(18) 두 파트를 실습했다. **17_LISTENER**는 Filter(서블릿 레벨) + Interceptor(Spring MVC 레벨) + Listener(Spring 이벤트)의 세 계층을 함께 구성하는 실습이다. Filter는 `FilterRegistrationBean`으로 URL 패턴과 실행 순서를 설정하고, Interceptor는 `HandlerInterceptor`의 `preHandle` / `postHandle` / `afterCompletion` 세 메서드로 컨트롤러 진입 전후와 뷰 렌더링 후를 각각 처리한다. Listener는 `ApplicationListener<E>`를 구현해 Spring 이벤트를 구독하며, 앱 시작 시 발생하는 `ContextRefreshedEvent`, 요청 완료 시 발생하는 `RequestHandledEvent`, 그리고 서비스 레이어에서 `ApplicationEventPublisher.publishEvent()`로 발행하는 커스텀 이벤트(`MemoAddEvent`) 세 가지를 실습했다. **18_HANDLER_MAPPING**은 `@RequestMapping` 어노테이션 없이 URL과 핸들러를 수동으로 매핑하는 세 가지 방식을 실습했다. `BeanNameUrlHandlerMapping`은 빈 이름을 URL로 사용하고, `SimpleUrlHandlerMapping`은 `Map<URL, 핸들러>`로 명시적으로 매핑하며 `setOrder()`로 우선순위를 제어한다. `RequestMappingHandlerMapping`은 Reflection으로 메서드를 선택하고 `RequestMappingInfo` 빌더로 요청 조건을 조합해 `registerMapping()`으로 등록하며, 이 방식은 `Controller` 인터페이스 없이 일반 클래스도 핸들러로 사용할 수 있다.

### 2) 세 계층 비교

| 계층 | 인터페이스 | 동작 범위 | 등록 방법 |
|------|----------|----------|----------|
| Filter | `jakarta.servlet.Filter` | DispatcherServlet 진입 전/후 (Servlet 레벨) | `FilterRegistrationBean` |
| Interceptor | `HandlerInterceptor` | 컨트롤러 전/후, 뷰 렌더링 후 (Spring MVC 레벨) | `WebMvcConfigurer.addInterceptors()` |
| Listener | `ApplicationListener<E>` | Spring 이벤트 발생 시점 | `@Bean` 등록 |

### 3) MemoFilter — 서블릿 Filter

```java
// @WebFilter("/memo/*")  ← 주석 처리, FilterConfig 방식 사용
@Slf4j
public class MemoFilter implements Filter {

    @Override
    public void doFilter(ServletRequest request, ServletResponse response,
                         FilterChain chain) throws IOException, ServletException {
        HttpServletRequest r = (HttpServletRequest) request;
        log.info("[FILTER] MemoFilter START " + r.getRequestURI());   // 전처리

        chain.doFilter(request, response);   // 다음 Filter 또는 Servlet으로 전달

        log.info("[FILTER] MemoFilter END");   // 후처리
    }
}
```

```java
// FilterConfig.java — FilterRegistrationBean으로 등록
@Configuration
public class FilterConfig {

    @Bean
    public FilterRegistrationBean<MemoFilter> memoFilter() {
        FilterRegistrationBean<MemoFilter> registrationBean = new FilterRegistrationBean<>();
        registrationBean.setFilter(new MemoFilter());
        registrationBean.addUrlPatterns("/memo/*");   // 적용 URL 패턴
        registrationBean.setOrder(1);                 // 실행 순서 (낮을수록 먼저)
        return registrationBean;
    }
}
```

> `@WebFilter` vs `FilterRegistrationBean` — Bean 방식은 `setOrder()`로 다중 필터 실행 순서 제어 가능

### 4) MemoInterceptor — HandlerInterceptor

```java
@Component @Slf4j
public class MemoInterceptor implements HandlerInterceptor {

    @Override
    public boolean preHandle(HttpServletRequest request, HttpServletResponse response,
                              Object handler) throws Exception {
        // 컨트롤러 진입 전 — false 반환 시 요청 차단
        log.info("[INTERCEPTOR] Memo컨트롤러 실행 전 : " + request.getRequestURI());
        return true;
    }

    @Override
    public void postHandle(HttpServletRequest request, HttpServletResponse response,
                           Object handler, ModelAndView modelAndView) throws Exception {
        // 컨트롤러 완료 후, 뷰 렌더링 전
        log.info("[INTERCEPTOR] Memo컨트롤러 실행 후 : " + request.getRequestURI());
    }

    @Override
    public void afterCompletion(HttpServletRequest request, HttpServletResponse response,
                                Object handler, Exception ex) throws Exception {
        // 뷰 렌더링 완료 후 (응답 전송 완료)
        log.info("[INTERCEPTOR] Memo컨트롤러 요청-응답 완료 후 : " + request.getRequestURI());
    }
}
```

```java
// WebMvcConfig.java — addInterceptors()로 등록
@Autowired
MemoInterceptor memoInterceptor;

@Override
public void addInterceptors(InterceptorRegistry registry) {
    registry.addInterceptor(memoInterceptor)
            .addPathPatterns("/memo/**")
            .excludePathPatterns("resources/css/**", "resources/js/**");
}
```

| 메서드 | 실행 시점 | 반환값 |
|--------|----------|--------|
| `preHandle` | 컨트롤러 실행 전 | `true`: 진행 / `false`: 차단 |
| `postHandle` | 컨트롤러 실행 후, 뷰 렌더링 전 | void |
| `afterCompletion` | 뷰 렌더링 완료 후 | void |

### 5) Listener — Spring 이벤트 시스템

#### Spring 기본 이벤트

```java
// 앱 컨텍스트 초기화 완료 시 자동 발생
public class C01CustomContextRefreshedListener
        implements ApplicationListener<ContextRefreshedEvent> {

    @Override
    public void onApplicationEvent(ContextRefreshedEvent event) {
        System.out.println("C01...onApplicationEvent invoke..." + event);
    }
}
```

```java
// DispatcherServlet이 HTTP 요청 처리 완료 후 자동 발생
public class C02RequestHandledEventListener
        implements ApplicationListener<RequestHandledEvent> {

    @Override
    public void onApplicationEvent(RequestHandledEvent event) {
        DispatcherServlet d = (DispatcherServlet) event.getSource();
        System.out.println("C02...onApplicationEvent invoke..." + d);
    }
}
```

| 이벤트 | 발생 시점 |
|--------|----------|
| `ContextRefreshedEvent` | 앱 시작 시 Spring 컨텍스트 초기화 완료 |
| `RequestHandledEvent` | DispatcherServlet HTTP 요청 처리 완료 |

#### 커스텀 이벤트

```java
// 이벤트 객체 — ApplicationEvent 상속 필수
public class MemoAddEvent extends ApplicationEvent {

    private MemoDTO dto;

    public MemoAddEvent(Object source, MemoDTO dto) {
        super(source);   // 이벤트 발행자(this) 전달
        this.dto = dto;
    }

    @Override
    public String toString() {
        return "MemoAddEvent{dto=" + dto + '}';
    }
}
```

```java
// 이벤트 리스너
public class MemoAddEventListener implements ApplicationListener<MemoAddEvent> {

    @Override
    public void onApplicationEvent(MemoAddEvent event) {
        System.out.println("[LISTENER] MEMO ADD EVENT " + event);
    }
}
```

```java
// 이벤트 발행 — MemoServiceImpl
@Service
public class MemoServiceImpl implements MemoService {

    @Autowired
    private ApplicationEventPublisher applicationEventPublisher;

    @Override
    @Transactional(rollbackFor = SQLException.class, transactionManager = "jpaTransactionManager")
    public boolean memoRegistration(MemoDTO memoDTO) throws Exception {
        memoDTO.setCreateAt(LocalDateTime.now());
        memoRepository.save(memoDTO.toEntity());
        applicationEventPublisher.publishEvent(new MemoAddEvent(this, memoDTO));   // 이벤트 발행
        return true;
    }
}
```

```java
// WebMvcConfig.java — Listener를 @Bean으로 등록
@Bean
public C01CustomContextRefreshedListener c01CustomContextRefreshedListener() {
    return new C01CustomContextRefreshedListener();
}
@Bean
public C02RequestHandledEventListener c02RequestHandledEventListener() {
    return new C02RequestHandledEventListener();
}
@Bean
public MemoAddEventListener memoAddEventListener() {
    return new MemoAddEventListener();
}
```

> `ApplicationListener<E>` 구현 클래스를 `@Bean`으로 등록하면 Spring이 자동으로 이벤트 구독 처리.  
> `publishEvent()`는 기본 동기 실행 — 리스너 완료 후 다음 코드 실행.

```

  │
  ▼

  │
  ▼

  │
  ▼

  ▼

  │
  ▼
뷰 렌더링 (redirect:/memo/list)
  │
  ▼
[INTERCEPTOR] afterCompletion()           ← 뷰 렌더링 후
  │
  ▼
[FILTER] MemoFilter.doFilter() — 후처리
  │
  ▼
응답 완료
  └── [LISTENER] C02RequestHandledEventListener.onApplicationEvent()
```

## Spring Boot — 18_HANDLER_MAPPING (HandlerMapping 커스터마이징)

---

### 1) 프로젝트 구조

```
18_HANDLER_MAPPING/src/main/java/com/example/demo/
└── Handler/
    ├── CustomHandler.java          # Controller 인터페이스 구현 핸들러
    └── RequestCustomHandler.java   # 일반 클래스 핸들러 (Reflection 매핑)
```

> 나머지 구조(Filter, Interceptor, Listener, Config)는 17_LISTENER와 동일. `WebMvcConfig`에 Handler 빈 등록 추가.

---

### 2) HandlerMapping 3가지 방식 개요

| 방식 | 매핑 기준 | 핸들러 조건 | URL |
|------|----------|-------------|-----|
| `BeanNameUrlHandlerMapping` | 빈 이름 = URL | `Controller` 인터페이스 구현 | `/custom_01` |
| `SimpleUrlHandlerMapping` | `Map<URL, 핸들러>` 명시 | `Controller` 인터페이스 구현 | `/custom_02` |
| `RequestMappingHandlerMapping` | `RequestMappingInfo` (Reflection) | 일반 클래스 (인터페이스 불필요) | `/custom_03`, `/custom_04` |
| 일반 `@RequestMapping` | 어노테이션 스캔 | `@Controller` 클래스 | 기존 방식 |

---

### 3) BeanNameUrlHandlerMapping — 빈 이름 = URL

```java
// WebMvcConfig.java
@Bean
BeanNameUrlHandlerMapping beanNameUrlHandlerMapping() {
    System.out.println("[HANDLER_MAPPER] beanNameUrlHandlerMapping init..");
    return new BeanNameUrlHandlerMapping();
}

@Bean(name = "/custom_01")   // 빈 이름이 곧 매핑 URL
public CustomHandler customHandler() {
    return new CustomHandler();
}
```

```java
// CustomHandler.java — Controller 인터페이스 구현 필수
public class CustomHandler implements Controller {

    @Override
    public ModelAndView handleRequest(HttpServletRequest request,
                                      HttpServletResponse response) throws Exception {
        System.out.println("[HANDLER] CustomHandler's handleRequest invoke..");
        ModelAndView modelAndView = new ModelAndView();
        modelAndView.setViewName("memo/add");
        return modelAndView;
    }
}
```

> `GET /custom_01` → 이름이 `/custom_01`인 빈을 찾아 `handleRequest()` 실행.  
> `Controller`는 `org.springframework.web.servlet.mvc.Controller` (Spring 자체 인터페이스).

---

### 4) SimpleUrlHandlerMapping — Map으로 명시적 매핑

```java
@Bean
SimpleUrlHandlerMapping simpleUrlHandlerMapping() {
    SimpleUrlHandlerMapping handlerMapping = new SimpleUrlHandlerMapping();

    Map<String, Object> map = new HashMap<>();
    map.put("/custom_02", new CustomHandler());   // URL → 핸들러 인스턴스 직접 지정

    handlerMapping.setUrlMap(map);
    handlerMapping.setOrder(0);   // 우선순위 (낮을수록 먼저 검색, 기본값 Integer.MAX_VALUE)

    return handlerMapping;
}
```

> `setOrder(0)` 설정으로 HandlerMapping 체인에서 가장 먼저 검색됨.  
> 정적 자원 처리도 기본적으로 `SimpleUrlHandlerMapping`이 담당.

---

### 5) RequestMappingHandlerMapping — Reflection 기반 수동 등록

```java
// RequestCustomHandler.java — 일반 클래스, 인터페이스 불필요
public class RequestCustomHandler {

    public String helloWorld() {
        System.out.println("[HANDLER] RequestCustomHandler's helloWorld invoke..");
        return "memo/add";   // 뷰 이름 반환
    }

    @ResponseBody
    public String helloWorld2() {
        System.out.println("[HANDLER] RequestCustomHandler's helloWorld2 invoke..");
        return "HELLOWORLD!!!!!!!";   // @ResponseBody → 응답 바디 직접 출력
    }
}
```

```java
// WebMvcConfig.java — /custom_03 : 뷰 반환
@Bean
RequestMappingHandlerMapping requestMappingHandlerMapping2() throws NoSuchMethodException {
    RequestMappingHandlerMapping handlerMapping = new RequestMappingHandlerMapping();

    Method method = RequestCustomHandler.class.getMethod("helloWorld", null);  // Reflection

    RequestMappingInfo info = RequestMappingInfo
            .paths("/custom_03")
            .methods(RequestMethod.GET)
            .build();

    handlerMapping.registerMapping(info, new RequestCustomHandler(), method);
    return handlerMapping;
}

// WebMvcConfig.java — /custom_04 : @ResponseBody JSON 응답
@Bean
RequestMappingHandlerMapping requestMappingHandlerMapping3() throws NoSuchMethodException {
    RequestMappingHandlerMapping handlerMapping = new RequestMappingHandlerMapping();

    Method method = RequestCustomHandler.class.getMethod("helloWorld2", null);

    RequestMappingInfo info = RequestMappingInfo
            .paths("/custom_04")
            .methods(RequestMethod.GET)
            .produces(MediaType.APPLICATION_JSON_VALUE)   // Accept 헤더 조건
            .build();

    handlerMapping.registerMapping(info, new RequestCustomHandler(), method);
    return handlerMapping;
}
```

> `RequestMappingInfo` 빌더 주요 메서드:
> - `.paths("/url")` — URL 패턴
> - `.methods(RequestMethod.GET)` — HTTP 메서드 조건
> - `.produces(MediaType.APPLICATION_JSON_VALUE)` — 응답 Content-Type 조건

### 6) 요청 처리 흐름

```
HTTP 요청
  │
  ▼
DispatcherServlet
  │
  ▼
HandlerMapping 체인 (order 순서대로 검색)
  ├── SimpleUrlHandlerMapping (order=0)    → /custom_02 매칭 → CustomHandler
  ├── BeanNameUrlHandlerMapping            → /custom_01 매칭 → CustomHandler
  └── RequestMappingHandlerMapping         → /custom_03, /custom_04 매칭 → RequestCustomHandler
  │
  ▼
HandlerAdapter가 핸들러 실행
  ├── /custom_01, /custom_02  → CustomHandler.handleRequest()  → ModelAndView("memo/add")
  ├── /custom_03              → helloWorld()                   → ModelAndView("memo/add")
  └── /custom_04              → helloWorld2() + @ResponseBody  → "HELLOWORLD!!!!!!!" 직접 출력
  │
  ▼
ViewResolver (/custom_04 제외)
  └── memo/add.html 렌더링
```

## Spring Boot — 19일 핵심 정리

---

### 1) Filter / Interceptor / Listener 핵심 포인트

| 항목 | Filter | Interceptor | Listener |
|------|--------|-------------|---------|
| 레이어 | Servlet 컨테이너 | Spring MVC | Spring 이벤트 시스템 |
| 인터페이스 | `Filter` → `doFilter()` | `HandlerInterceptor` → 3개 메서드 | `ApplicationListener<E>` → `onApplicationEvent()` |
| 등록 | `FilterRegistrationBean` | `addInterceptors()` | `@Bean` |
| 요청 차단 | `chain.doFilter()` 미호출 | `preHandle()` `return false` | 해당 없음 |
| 트리거 | 모든 서블릿 요청 | 매핑된 컨트롤러 경로 | 이벤트 `publishEvent()` 호출 |

---

### 2) HandlerMapping 3가지 핵심 포인트

| 방식 | 핵심 코드 | 특이사항 |
|------|---------|---------|
| `BeanNameUrl` | `@Bean(name="/url")` | 빈 이름 = URL, `Controller` 인터페이스 구현 필수 |
| `SimpleUrl` | `setUrlMap(map)` + `setOrder(0)` | `order` 값이 낮을수록 우선 검색 |
| `RequestMappingHandler` | `Reflection` + `RequestMappingInfo` + `registerMapping()` | 일반 클래스도 핸들러 가능, `@ResponseBody` 사용 가능 |

---

### 3) 커스텀 이벤트 패턴 정리

```
① 이벤트 클래스 정의
   class MemoAddEvent extends ApplicationEvent
   생성자: super(source) — 발행자 전달

② 리스너 정의
   class MemoAddEventListener implements ApplicationListener<MemoAddEvent>
   → @Bean으로 등록

③ 발행
   @Autowired ApplicationEventPublisher applicationEventPublisher
   applicationEventPublisher.publishEvent(new MemoAddEvent(this, dto))
   → 동기 실행 (기본값) — 리스너 완료 후 다음 코드 실행
```

### 4) RequestMappingInfo 빌더 패턴 정리

```java
Method method = SomeHandler.class.getMethod("메서드명", null);   // Reflection

RequestMappingInfo info = RequestMappingInfo
    .paths("/url")                                // URL 조건
    .methods(RequestMethod.GET)                   // HTTP 메서드 조건
    .produces(MediaType.APPLICATION_JSON_VALUE)   // 응답 타입 조건 (선택)
    .build();

handlerMapping.registerMapping(info, new SomeHandler(), method);  // 등록
```
