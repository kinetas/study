# 7월 6일 필기 — Docker: 이미지 빌드 → 컨테이너 네트워크 → docker-compose 오케스트레이션

## 핵심요약

`Docker/00_INIT → 01_IMAGES → 02_CONTAINER → 03_COMPOSE` 순서로 단계를 하나씩 올려가며 실습했다. 구성 서비스는 `BN`(Spring Boot 백엔드), `DB`(MySQL), `REDIS`, `fn`(React 프론트) 4개. **00_INIT**에서 Dockerfile 기본 문법(레이어 캐싱)을 익히고, **01_IMAGES**에서 각 서비스를 독립 이미지로 멀티스테이지 빌드했다. 이 상태에서는 `localhost`로 하드코딩된 접속 정보 때문에 컨테이너끼리 통신이 안 되는 걸 직접 확인했고, **02_CONTAINER**에서 커스텀 네트워크 + 컨테이너 이름을 호스트명으로 써서 연결하고 nginx 리버스 프록시로 프론트/백엔드를 하나의 진입점으로 묶었다. **03_COMPOSE**에서 이 모든 걸 `docker-compose.yml` 하나로 통합하면서 헬스체크, `depends_on(condition)`, named volume, 환경변수 오버라이드(Spring relaxed binding)까지 정리했다.

## 00_INIT — Dockerfile 기초

```dockerfile
# Node.js 18 공식 이미지를 베이스로 사용(Debian 기반, 풀 사이즈)
FROM node:18
WORKDIR /app
# package.json과 package-lock.json만 먼저 복사
COPY package*.json ./
RUN npm install
# 나머지 소스코드 전체 복사
COPY . .
CMD ["npm","start"]
```

| 명령 | 역할 |
|---|---|
| `FROM` | 베이스 이미지 지정 |
| `WORKDIR` | 이후 명령의 기준 디렉토리 (없으면 자동 생성) |
| `COPY package*.json ./` → `RUN npm install` → `COPY . .` | **레이어 분리가 핵심.** package.json이 안 바뀌면 `npm install` 레이어는 캐시 재사용 → 소스만 바뀐 재빌드가 훨씬 빨라짐 |
| `CMD` | 컨테이너 시작 시 실행되는 기본 명령 (1개만 유효, 마지막 것이 적용) |

## 01_IMAGES — 서비스별 이미지 빌드

각 서비스(BN/DB/REDIS/fn)를 독립된 이미지로 빌드하는 단계. **아직 서로 연결은 안 함.**

### BN (Spring Boot) — 멀티스테이지 빌드

```dockerfile
# 빌드 스테이지
FROM gradle:8.14.1-jdk21 AS build
WORKDIR /app
COPY . .
RUN gradle build --no-daemon -x test

# 실행 스테이지
FROM eclipse-temurin:21-jdk-alpine
WORKDIR /app
COPY --from=build /app/build/libs/*.jar app.jar
EXPOSE 8080
CMD ["java", "-jar", "app.jar"]
```

- 1단계: `gradle` 이미지(무거움, 빌드 도구 포함)에서 jar 생성
- 2단계: `eclipse-temurin:21-jdk-alpine`(실행 전용, 가벼움)에 **jar 결과물만** 복사
- → **빌드 도구와 실행 환경을 분리**해서 최종 이미지 용량을 줄이는 표준 패턴 (`COPY --from=build`)

### fn (React) — 멀티스테이지 빌드

```dockerfile
### 1단계: React 앱 빌드 ###
FROM node:20-alpine AS build
WORKDIR /app
COPY package*.json ./
RUN npm install
COPY . .
RUN npm run build

### 2단계: Nginx 이미지로 정적 파일 배포 ###
FROM nginx:alpine
COPY --from=build /app/build /usr/share/nginx/html
EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]
```

- React는 빌드하면 정적 파일(html/js/css)만 남기 때문에, 실행 단계는 Node가 아니라 **nginx**로 서빙 → 이미지가 훨씬 가벼워짐

### DB / REDIS — 단일 스테이지

```dockerfile
# DB Dockerfile
FROM mysql:8.0
ENV MYSQL_ROOT_PASSWORD=your_password
ENV MYSQL_DATABASE=bookdb
ENV MYSQL_USER=dbconn
ENV MYSQL_PASSWORD=your_password
EXPOSE 3306
CMD ["mysqld"]
```

```dockerfile
# REDIS Dockerfile
FROM redis:latest
WORKDIR /usr/local/etc/redis
EXPOSE 6479
CMD ["redis-server"]
```

> REDIS의 `EXPOSE 6479`는 오타 — 실제 redis 리스닝 포트는 6379. `EXPOSE`는 문서화 목적이라 컨테이너 동작 자체엔 영향 없지만, 나중에 포트 매핑할 때 헷갈리기 쉬움.

### 문제점 — 컨테이너끼리 서로를 모른다

이 단계는 각 이미지를 `docker run`으로 **따로** 띄우는 걸 전제로 함. 그런데 BN의 설정이 이렇게 되어 있었다:

```java
// DataSourceConfig.java (수정 전)
dataSource.setDriverClassName("com.mysql.cj.jdbc.Driver");
dataSource.setJdbcUrl("jdbc:mysql://localhost:3306/testdb");
dataSource.setUsername("root");
dataSource.setPassword("1234");
```

```properties
# application.properties (수정 전)
spring.redis.host=localhost
```

컨테이너 안에서 `localhost`는 **자기 자신**을 가리킨다. BN 컨테이너 입장에서 `localhost:3306`은 DB 컨테이너가 아니라 BN 컨테이너 자신의 3306 포트 → 당연히 연결 실패. **컨테이너 간 통신은 `localhost`가 아니라 네트워크로 묶어서 이름으로 찾아야 한다**는 걸 에러로 직접 확인.

## 02_CONTAINER — 컨테이너 연결 (docker network)

### 하드코딩 제거 → `@Value`로 외부 설정 주입

```java
// DataSourceConfig.java (수정 후)
@Configuration
public class DataSourceConfig {
    @Value("${spring.datasource.classname}") private String DBClassName;
    @Value("${spring.datasource.url}")       private String DBJdbcUrl;
    @Value("${spring.datasource.username}")  private String username;
    @Value("${spring.datasource.password}")  private String password;

    @Bean
    public DataSource dataSource() {
        HikariDataSource dataSource = new HikariDataSource();
        dataSource.setDriverClassName(DBClassName);
        dataSource.setJdbcUrl(DBJdbcUrl);
        dataSource.setUsername(username);
        dataSource.setPassword(password);
        return dataSource;
    }
}
```

```properties
# application.properties (수정 후) — 컨테이너 이름을 호스트명으로 사용
spring.redis.host=redis-container
spring.datasource.classname=com.mysql.cj.jdbc.Driver
spring.datasource.url=jdbc:mysql://db-container:3306/bookdb
spring.datasource.username=root
spring.datasource.password=your_password
```

- 코드에 값을 박아두지 않고 `application.properties`(나중엔 환경변수)에서 주입받도록 리팩터링 → 환경이 바뀌어도 **코드/이미지 재빌드 없이** 설정만 바꾸면 됨
- `localhost` → `db-container` / `redis-container` : 같은 커스텀 네트워크에 올라간 컨테이너는 **컨테이너 이름 = 호스트명**으로 서로를 찾을 수 있음 (Docker의 내장 DNS)

### 헬스체크를 위한 curl 설치

```dockerfile
# BN Dockerfile 추가분
RUN apk add --no-cache curl
```

- alpine 베이스 이미지는 기본적으로 `curl`이 없음 → 컨테이너 안에서 `curl`로 헬스체크를 하려면 **이미지 자체에 curl을 설치**해둬야 함

### fn/nginx — 정적 서빙 + 리버스 프록시

```nginx
events {
    worker_connections 1024;
}

http {
    server {
        listen       80;
        server_name  localhost;

        # ① API 요청 → Spring Boot 컨테이너로 프록시
        location /bn/ {
            proxy_pass http://bn-container:8080/;
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
            proxy_set_header X-Forwarded-Proto $scheme;
        }

        # ② React 정적 파일 서빙 (SPA 라우팅)
        location / {
            root   /usr/share/nginx/html;
            index  index.html;
            try_files $uri /index.html;
        }

        error_page 500 502 503 504 /50x.html;
        location = /50x.html { root /usr/share/nginx/html; }
    }
}
```

```env
# fn/.env (수정 전 → 수정 후)
# REACT_APP_API_URL=http://bn-container:8080   ← 브라우저는 이 컨테이너 이름을 모름
REACT_APP_API_URL=/bn
```

| 포인트 | 설명 |
|---|---|
| `proxy_pass http://bn-container:8080/` | nginx 컨테이너와 bn 컨테이너가 같은 네트워크에 있어야 컨테이너 이름으로 통신 가능 |
| `proxy_set_header` 4종 | 원 요청의 Host/실제 IP/프록시 경유 IP/프로토콜을 백엔드에 전달 (로그·인증용) |
| `try_files $uri /index.html` | React Router 새로고침 시 404 방지 (SPA 표준 설정) |
| `REACT_APP_API_URL=/bn` | **브라우저(클라이언트)는 Docker 네트워크 안에 없기 때문에** `bn-container`라는 이름을 절대 알 수 없음. 그래서 프론트가 백엔드를 직접 호출하지 않고, 자신을 서빙하는 nginx에게 상대경로로 요청 → nginx가 내부망에서 `/bn/`을 백엔드로 프록시 |

## 03_COMPOSE — docker-compose로 오케스트레이션

```yaml
version: '3.8'

networks:
  custom-network:
    driver: bridge
    ipam:
      config:
        - subnet: 172.30.0.0/24
          gateway: 172.30.0.1

services:
  db:
    image: db
    build: { context: ./DB, dockerfile: Dockerfile }
    container_name: db-container
    networks:
      custom-network: { ipv4_address: 172.30.0.10 }
    healthcheck:
      test: ["CMD", "mysqladmin", "ping", "-h", "127.0.0.1"]
      interval: 30s
      timeout: 10s
      retries: 5
    ports: ["3330:3306"]
    environment:
      MYSQL_ROOT_PASSWORD: your_password
      MYSQL_DATABASE: testdb
      MYSQL_USER: dbconn
      MYSQL_PASSWORD: your_password
      MYSQL_ROOT_HOST: '%'
    volumes:
      - db-data:/var/lib/mysql

  redis:
    image: redis
    build: { context: ./REDIS }
    container_name: redis-container
    ports: ["6380:6379"]
    networks:
      custom-network: { ipv4_address: 172.30.0.20 }

  bn:
    image: bn
    build: { context: ./BN }
    container_name: bn-container
    ports: ["8080:8080"]
    environment:
      SPRING_DATASOURCE_URL: jdbc:mysql://db-container:3306/testdb
      SPRING_DATASOURCE_USERNAME: root
      SPRING_DATASOURCE_PASSWORD: your_password
      SPRING_REDIS_HOST: redis
      SERVER_PORT: 8080
    healthcheck:
      test: ["CMD-SHELL", "curl -f http://127.0.0.1:8080/actuator/health"]
      interval: 30s
      timeout: 10s
      retries: 5
    depends_on:
      db: { condition: service_healthy }
    networks:
      custom-network: { ipv4_address: 172.30.0.40 }

  fn:
    image: fn
    build: { context: ./FN }
    container_name: fn-container
    ports: ["3000:80"]
    depends_on:
      bn: { condition: service_healthy }
    networks:
      custom-network: { ipv4_address: 172.30.0.30 }

volumes:
  db-data:
```

### 핵심 포인트

| 항목 | 설명 |
|---|---|
| `networks.custom-network.ipam` | 서브넷/게이트웨이를 직접 지정하고, 서비스별로 `ipv4_address` 고정 (db `.10`, redis `.20`, fn `.30`, bn `.40`) |
| `healthcheck` | 컨테이너가 "떠 있음"과 "정상 동작함"은 다름. db는 `mysqladmin ping`, bn은 `curl .../actuator/health`로 실제 준비 상태를 체크 |
| `depends_on: { condition: service_healthy }` | 단순 시작 순서(`depends_on`)만으로는 부족 — DB가 완전히 뜨기 전에 BN이 붙으면 연결 실패. **healthy가 될 때까지 대기**하도록 조건을 걸어야 함 |
| `volumes: db-data:/var/lib/mysql` | named volume으로 DB 데이터 영속화 — 컨테이너를 지워도 데이터는 남음 |
| `actuator/health` | 헬스체크 대상으로 쓰려면 BN에 Spring Boot Actuator 의존성 + `management.endpoint.health` 노출 설정 필요 |

### 환경변수가 이미지 안 설정보다 우선 적용됨 (Spring relaxed binding)

`docker-compose.yml`의 `environment`에 적은 값이 이미지 안 `application.properties`보다 **우선 적용**된다.

```
환경변수 (대문자 + 언더스코어)       properties 키 (소문자 + dot)
SPRING_DATASOURCE_URL          ↔   spring.datasource.url
SPRING_REDIS_HOST              ↔   spring.redis.host
```

- Spring Boot가 이 규칙(relaxed binding)으로 환경변수를 자동 매핑해줌
- 그래서 **이미지를 다시 빌드하지 않고도** compose 파일만 바꿔서 DB 접속 정보 등을 바꿀 수 있음
- 실제로 compose의 db 환경변수는 `testdb`/`dbconn`인데 BN 이미지 기본값(`application.properties`)은 `bookdb`/`root` — compose의 `SPRING_DATASOURCE_URL: jdbc:mysql://db-container:3306/testdb`가 우선 적용되어 동작하지만, 두 값이 서로 다른 채로 방치되어 있어 헷갈리기 쉬움 (나중에 하나로 통일 필요)

## 6일 핵심 정리

---

### 1) 이미지 빌드 vs 컨테이너 실행 — 단계별 차이

```
01_IMAGES   : 이미지만 개별 빌드 (서로 모름, localhost 하드코딩 → 연결 실패)
02_CONTAINER: docker network + 컨테이너 이름을 호스트명으로 사용 (수동 연결)
03_COMPOSE  : docker-compose.yml 하나로 network/healthcheck/순서/볼륨까지 선언적으로 관리
```

### 2) 컨테이너 간 통신 규칙

```
같은 네트워크에 있어야만 컨테이너 이름으로 서로를 찾을 수 있다 (Docker 내장 DNS)
  BN → DB     : jdbc:mysql://db-container:3306/...
  BN → Redis  : spring.redis.host=redis-container
  nginx → BN  : proxy_pass http://bn-container:8080/

브라우저(클라이언트)는 Docker 네트워크 밖에 있으므로 컨테이너 이름을 모른다
  → 프론트는 백엔드를 직접 호출하지 않고, 자신을 서빙하는 nginx에 상대경로(/bn)로 요청
  → nginx가 내부망에서 프록시
```

### 3) docker-compose 핵심 문법

```
networks.ipam          → 서브넷/게이트웨이 직접 지정 + 서비스별 고정 IP
healthcheck             → "떠 있음"이 아니라 "정상 동작"을 체크 (mysqladmin ping, curl actuator/health)
depends_on.condition    → service_healthy 를 걸어야 healthcheck 통과까지 실제로 대기함
volumes(named volume)  → 컨테이너 삭제와 무관하게 데이터 영속화
environment             → 이미지 내 설정보다 우선 적용 (Spring: SPRING_X_Y ↔ spring.x.y 자동 매핑)
```
