## 16일 핵심 정리

---

### 1) 오염 체크: "몇 %가 숫자로 바뀌나"로 오염 컬럼 자동 판별

```python
for c in train.select_dtypes(include='object').columns:
    ratio = pd.to_numeric(train[c].astype(str).str.replace('_',''), errors='coerce').notna().mean()
    # ratio >= 0.9 이면 사실상 숫자 컬럼인데 '_' 같은 잡음이 섞여 문자열로 저장된 것
```
전수조사 대신 **변환 성공률**로 판단하니 26개 컬럼을 눈으로 다 훑지 않고도 오염 컬럼 8개를 한 번에 찾아냄. 임계값 90%는 "일부 진짜 범주형(주소, 이름처럼 우연히 숫자가 많은 값)"을 오염으로 잘못 판정하지 않기 위한 여유값.

### 2) 결측치가 "늦게" 드러나는 순서를 실제로 확인

```
오염 수정 전: train.isnull().sum().sum() → 0 (전처리 안 한 게 아니라 확인해보니 없었던 상태)
오염 수정 후: 62162 (문자열 → 숫자 변환 실패분이 NaN으로 새로 생김)
```
15일엔 `cat_cols`/`num_cols`가 빈 리스트로 나온 게 "실행 순서 뒤섞임 때문"이라고 의심했는데, 16일 재실행으로 보니 실제 원인은 **오염 수정을 먼저 해야 결측이 드러나는 순서 문제**였음. `null_series[(null_series>0) & (train.dtypes=='object')]`처럼 조건을 boolean mask로 걸어 필터링하면 리스트컴프리헨션보다 dtype 변경 후 결과가 더 안정적으로 잡힘.

### 3) 원핫 대신 라벨 인코딩을 쓰는 기준

```
이진분류(Binary_Classfication) : pd.get_dummies(전체 concat) — 카테고리 수 적당 (Adult Census)
다중분류(Multiclass_Classfication) : LabelEncoder — Name/SSN 등 카테고리 수가 너무 많음
```
`get_dummies`는 카테고리 하나당 열 하나가 생기므로, `Name`처럼 사실상 거의 고유값에 가까운 컬럼에 쓰면 열 개수가 감당 안 되게 늘어남. train+test를 합쳐서 `fit`하는 원칙(인코딩 기준 통일)은 두 방식 모두 동일.

### 4) macro vs weighted f1 — 숫자 하나가 아니라 "격차"를 봐야 함

```
f1(macro)    : 0.7751  — 클래스마다 동일 가중치(1표씩)
f1(weighted) : 0.7883  — 표본 수로 가중평균 (Standard가 제일 많아 유리)
격차 0.0132  → Good(소수 클래스, support 3566)의 f1이 0.72로 상대적으로 낮다는 신호
```
정확도나 f1 하나만 보고 "0.79면 괜찮네"로 끝내면 소수 클래스가 잘 안 맞는다는 사실을 놓침. macro와 weighted를 같이 찍어 그 차이로 불균형 영향을 읽는 습관이 필요.

### 5) 모델 저장 검증은 "저장했다"가 아니라 "재로드해서 예측이 같다"까지

```python
joblib.dump(rf, 'model.pkl')
loaded = joblib.load('model.pkl')
(loaded.predict(X_val) == pred).all()   # True
```
파일이 생겼다고 저장이 맞게 됐다는 보장은 없음 — 재로드 후 기존 예측과 비교하는 한 줄을 반드시 같이 넣어야 저장 자체의 무결성을 확인한 게 됨.
