# 7월 10일 필기 — Docker: 배포용 compose 분리 + Jenkins CI/CD 컨테이너 통합

## 핵심요약

`Docker/04_DOCKER_COMPOSE_DEPLOY` 단계. 03_COMPOSE에서 완성한 db/redis/bn/fn 4개 서비스 구성은 그대로 두고, **compose 파일을 배포용(`docker-compose.yml`)과 로컬 검증용(`docker-compose-local.yml`)으로 분리**했다. 로컬용 compose에는 **Jenkins 컨테이너**를 추가해서 CI/CD 파이프라인까지 같은 네트워크에서 띄워볼 수 있게 구성했고, Jenkins 컨테이너 안에서 호스트의 Docker를 직접 제어할 수 있도록 `docker.sock`을 마운트했다. 또한 BN의 `application.properties` 기본값 자체를 컨테이너 이름(`db-container`/`redis-container`) 기준으로 정리해서, 환경변수 오버라이드가 없어도 같은 네트워크 안에서 바로 붙게 만들었다.

---

## compose 파일을 용도별로 분리

```
docker-compose.yml        : db, redis, bn, fn            → 실제 배포(운영) 대상
docker-compose-local.yml  : db, redis, bn, fn, jenkins    → 로컬에서 CI/CD까지 함께 검증
```

- 운영 환경에는 필요 없는 CI/CD 도구(Jenkins)를 배포용 compose에서 빼고, 로컬 개발/검증용 compose에만 포함
- 서비스 이름·네트워크·볼륨 구조는 두 파일이 동일 — `jenkins` 서비스 하나만 로컬용에 추가된 차이

---

## Jenkins 서비스 추가

```yaml
jenkins:
  image: jenkins
  build:
    context: ./JENKINS
  container_name: jenkins-container
  ports:
    - "9090:8080"
    - "50000:50000"
  volumes:
    - jenkins-data:/var/jenkins_home
    - /var/run/docker.sock:/var/run/docker.sock
  networks:
    - custom-network
```

| 항목 | 설명 |
|---|---|
| `9090:8080` | Jenkins 웹 UI. 호스트 8080은 BN이 이미 점유하고 있어서 9090으로 매핑 |
| `50000:50000` | Jenkins 에이전트(빌드 노드) 연결용 포트 |
| `jenkins-data:/var/jenkins_home` | named volume으로 Jenkins 설정/빌드 이력을 영속화 (컨테이너를 지워도 파이프라인 설정은 남음) |
| `/var/run/docker.sock:/var/run/docker.sock` | 호스트의 Docker 데몬 소켓을 그대로 컨테이너에 마운트 |

### `docker.sock` 마운트가 하는 일 — Docker-outside-of-Docker

컨테이너 안에서 `docker` 명령을 실행하면 보통은 컨테이너 안에 Docker 데몬이 따로 있어야 하는데(Docker-in-Docker), 대신 **호스트의 소켓을 그대로 공유**하면 컨테이너 안에서 실행한 `docker build`/`docker run`이 컨테이너 자신이 아니라 **호스트의 Docker 데몬**에 명령을 전달한다.

```
Jenkins 컨테이너 --(docker.sock 공유)--> 호스트 Docker 데몬 --> 이미지 빌드/컨테이너 실행
```

- 장점: 컨테이너 안에 무거운 Docker 데몬을 새로 띄울 필요가 없음
- 주의점: 이 소켓 자체가 호스트에 대한 사실상의 루트 권한이라, Jenkins 컨테이너(혹은 그 안에서 도는 파이프라인 스크립트)가 뚫리면 호스트 전체가 위험해질 수 있음 → 운영 반영 전에는 권한 분리(전용 빌드 에이전트, rootless docker 등) 검토가 필요

## Jenkins Dockerfile — CLI 도구 설치

```dockerfile
FROM jenkins/jenkins:lts
USER root

RUN apt-get update && apt-get -y install --no-install-recommends \
    curl unzip gnupg ca-certificates && rm -rf /var/lib/apt/lists/*

# Docker CLI 설치 (데몬은 호스트 소켓 마운트로 이미 확보되어 있으므로 CLI만 있으면 됨)
RUN curl -fsSL https://get.docker.com | sh

# AWS CLI 설치 (빌드 산출물을 AWS로 배포하기 위함)
RUN curl "https://awscli.amazonaws.com/awscli-exe-linux-x86_64.zip" -o "awscliv2.zip" \
    && unzip awscliv2.zip && ./aws/install && rm -rf awscliv2.zip aws

USER jenkins
```

- 공식 `jenkins/jenkins:lts` 이미지에는 Docker CLI가 없음 → 파이프라인에서 `docker build`/`docker push`를 쓰려면 이미지에 CLI를 직접 설치해야 함 (데몬은 `docker.sock` 마운트로 이미 연결돼 있으니 CLI 바이너리만 있으면 됨)
- AWS CLI까지 설치 → 빌드한 이미지/아티팩트를 AWS(ECR, S3, EC2 등)로 배포하는 파이프라인을 염두에 둔 구성
- 패키지 설치는 `root`로, 실제 Jenkins 실행은 다시 `jenkins` 유저로 복귀 — 필요한 구간에서만 권한을 올리는 최소 권한 원칙

## BN 설정 — 이미지 기본값 자체를 컨테이너 네트워크 기준으로 정리

```properties
# application.properties (04 단계)
spring.redis.host=redis-container
spring.redis.port=6379
spring.datasource.classname=com.mysql.cj.jdbc.Driver
spring.datasource.url=jdbc:mysql://db-container:3306/testdb
spring.datasource.username=root
spring.datasource.password=your_password
```

- 02_CONTAINER 단계에서는 `@Value`로 **외부 주입만** 받도록 리팩터링했고 이미지 기본값 자체는 그대로 남아 있었는데, 04 단계부터는 **이미지 안 기본 설정도 컨테이너 이름 기준**으로 정리
- 그 결과 compose의 `environment` 오버라이드가 없어도 같은 커스텀 네트워크 위에서는 바로 연결됨 (오버라이드는 "필요하면 덮어쓰는" 안전장치로 남고, 기본값 자체가 이미 맞는 상태)
- DB 이름도 `testdb`/`dbconn`으로 compose 쪽과 통일 — 03 단계 정리 노트에 남아있던 `bookdb`/`root` 불일치 이슈를 여기서 해소함

## FN — 로컬/배포 전환 지점 표시

```env
# REACT_APP_API_URL=http://localhost:8080
REACT_APP_API_URL=/bn
# REACT_APP_API_URL=http://<EC2-IP>:8080
```

- 로컬 개발: nginx가 `/bn` 경로를 백엔드로 프록시 (02_CONTAINER에서 정리한 구조 그대로)
- 실제 배포: EC2 등 운영 서버 주소로 바꿔서 빌드하는 지점을 주석으로 남겨 로컬/배포 설정 전환을 명시적으로 표시

## 10일 핵심 정리

---

### 1) compose 파일 분리 전략

```
docker-compose.yml       → 배포(운영)에 필요한 서비스만: db / redis / bn / fn
docker-compose-local.yml → 로컬 검증용, CI/CD 도구까지 포함: db / redis / bn / fn / jenkins
```

### 2) Jenkins 컨테이너로 CI/CD 붙이기

```
docker.sock 마운트  → 컨테이너 안에서 호스트 Docker를 직접 제어 (Docker-outside-of-Docker)
Docker CLI 설치     → 파이프라인에서 docker build/push 실행 가능
AWS CLI 설치        → 빌드 결과물을 AWS로 배포 가능
jenkins-data volume → 파이프라인/설정을 컨테이너 재생성과 무관하게 유지
```

### 3) 설정 정리 원칙

```
이미지 기본값(application.properties) 자체를 컨테이너 네트워크 기준으로 맞춰둔다
  → compose environment는 "덮어쓰기용 안전장치"로만 남기고, 기본 상태에서도 바로 동작하게
DB 이름/계정 등은 compose와 이미지 기본값을 하나로 통일해서 혼동 요소를 없앤다
.env / .properties의 배포용 값은 주석으로 남겨 로컬 ↔ 배포 전환 지점을 명시한다
```

### 4) 남은 과제

```
docker.sock 마운트는 호스트 루트 권한과 사실상 동급 → 운영 반영 전 권한 분리 검토
Jenkinsfile(파이프라인 코드)이 아직 저장소에 없음 → 다음 단계에서 코드화 필요
DB 비밀번호/OAuth 시크릿이 소스에 하드코딩된 상태 → .env + .gitignore로 분리 필요
```
