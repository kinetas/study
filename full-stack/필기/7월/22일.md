# 7월 22일 필기 — 머신러닝: 로지스틱 회귀(질병 예측·유방암 진단), 성능 개선(교차검증·GridSearchCV·파이프라인)

## 핵심요약

**04_MACHINE_LEARNING/Supervised_Learning/04_Logistic_Regression/01.ipynb**로 건강검진 데이터(`health_survey.csv`)를 이용한 로지스틱 회귀 이진분류를 실습했다. 시그모이드로 "선형 값을 0~1 확률로 눌러 담는" 원리를 직접 그려 확인하고, `StandardScaler`는 train에만 `fit`해야 한다는 원칙을 재확인했다. 가장 인상적인 실측은 **스케일러를 저장·재사용하지 않으면 새 환자의 질병확률이 0.806(정상)에서 1.000(스케일러 누락)으로 완전히 틀어진다**는 것 — 모델만 저장하고 스케일러를 빠뜨리는 실수가 실제로 어떤 결과를 낳는지 숫자로 봤다. 이어서 연습문제로 유방암(Breast Cancer) 진단 데이터에 같은 파이프라인을 적용했는데, 여기서 **의학의 "양성(benign)"과 머신러닝의 "양성(positive=1)"이 뜻이 정반대**라는 함정과, 정확도 98%에 숨어있던 **"악성을 놓친(FN)" 방향으로만 쏠린 오류**를 혼동행렬로 잡아내는 과정이 이 노트북 전체의 핵심이었다. 이어서 **05_Performance_Improvements/01.ipynb**에서는 당뇨병 데이터로 "한 번 분할해서 검증"하는 방식의 불안정성을 짚고, `cross_val_score`(5-fold 교차검증)로 더 신뢰할 수 있는 점수를 얻은 뒤, `GridSearchCV`로 튜닝한 "최적 CV 점수"가 실제 검증셋 점수와 크게 다를 수 있다는 **점수 착시**를 직접 확인했다. 와인 품질 연습문제에서는 `RandomForestRegressor`를 튜닝해봤지만 개선폭이 0.006 수준으로 미미해, 튜닝 비용(후보 수 × fold 수) 대비 효과가 항상 크지는 않다는 트레이드오프까지 정리했다.

| 단계 | 코드 | 핵심 |
|---|---|---|
| EDA | `df['disease'].value_counts()`(1:588, 0:412), `df.corr()['disease']`, `groupby(pd.cut(...))` | 상관 1위 `bmi`(0.21)로 회귀 때보다 약함. 나이대·흡연·활동수준별 그룹집계로 나이↑·흡연·활동↓일수록 질병 비율이 오르는 경향 확인 |
| 시그모이드 원리 | `z = np.linspace(-8,8,100)`; `sigmoid = 1/(1+np.exp(-z))` | 선형 값을 0~1 확률로 눌러 담는 함수를 직접 그려서 z=0일 때 정확히 0.5임을 확인 |
| 스케일링 | `StandardScaler().fit_transform(X_tr)` → `transform(X_val)` | fit은 train에만, val은 transform만 — 데이터 누수 방지 원칙이 회귀·분류 가리지 않고 동일 |
| 학습·평가 | `LogisticRegression().fit(X_tr_s, y_tr)` → `accuracy_score`/`roc_auc_score` | 정확도 0.615, ROC-AUC 0.6271 — 특성 4개로는 질병 여부 설명이 부족한 약한 모델 |
| 계수 해석 | `pd.Series(model.coef_[0], index=X.columns).sort_values(...)` | `bmi` 0.436(위험↑ 최대) > `age` 0.359 > `smoker` 0.116 > `activity_level` -0.243(활동 많을수록 위험↓) |
| 임계값 튜닝 | `for t in [0.5, 0.3]: pt = (proba>=t).astype(int)` | 0.5→recall 0.7373/precision 0.6541, 0.3→recall 1.0000/precision 0.6051 — 임계값 낮추면 recall↑·precision↓ 트레이드오프 |
| 모델 내보내기 | `joblib.dump(model,...)` + **`joblib.dump(scaler,...)`** | 스케일러 누락 시 새 환자 질병확률이 0.806 → 1.000으로 완전히 틀어짐 — 스케일러도 반드시 같이 저장 |

## 04_MACHINE_LEARNING — Supervised_Learning · 성능 개선 (`05_Performance_Improvements/01.ipynb`)

| 단계 | 코드 | 핵심 |
|---|---|---|
| 1. 단일 분할의 한계 | `train_test_split` → `LinearRegression` → `r2_score` | 단일 분할 R² 0.3322 — 어떤 20%가 뽑혔는지에 따라 흔들릴 수 있는 값 |
| 2. 교차검증 | `cross_val_score(LinearRegression(), X, y, cv=5, scoring='r2')` | 5개 점수 `[0.43,0.523,0.483,0.426,0.55]` 평균 R² 0.4823 — 단일 분할보다 안정적 |
| 3. 규제 모델 비교 | `Ridge(alpha=1)` vs `Lasso(alpha=0.1)` | Ridge 0.4102, Lasso 0.4795 — 같은 교차검증 틀 위에서 모델만 바꿔 공정 비교 |
| 4. 하이퍼파라미터 튜닝 | `GridSearchCV(Ridge(), {'alpha':[0.01,0.1,1,10,100]}, cv=5)` | 최적 alpha 0.01, 최적 CV R² 0.5267 — 그런데 실제 `X_val` 검증 R²는 0.33으로 뚝 떨어짐(점수 착시) |
| 5. 파이프라인 | `make_pipeline(StandardScaler(), Ridge(alpha=1))` | 평균 R² 0.4822 — 매 fold의 train에만 스케일링 fit되어 누수 없이 얻은 정직한 점수 |

## 22일 핵심 정리

---

### 1) 모델만 저장하고 스케일러를 빠뜨리면 예측이 완전히 틀어진다

```python
loaded_model.predict_proba(loaded_scaler.transform(new))[0, 1]   # 0.806 (정상)
loaded_model.predict_proba(new)[0, 1]                             # 1.000 (스케일러 누락)
```
학습 때 표준화된 좌표계를 기준으로 계수를 배운 모델에 원본 스케일 값을 그대로 넣으면 값이 엉뚱하게 커져서 확률이 극단으로 쏠린다. `joblib.dump(model, ...)` 옆에 `joblib.dump(scaler, ...)`를 반드시 같이 써야 하는 이유가 숫자로 확인됨.

### 2) "양성(positive)"이라는 단어를 코드에 앞서 확인해야 하는 이유

```python
print(data.target_names)   # ['malignant', 'benign']  ->  1 = benign(건강한 쪽)
```
의학에서 "양성"은 괜찮다는 뜻이지만 sklearn의 "양성 클래스(1)"는 그냥 라벨 1일 뿐이다. 이 데이터에서는 두 뜻이 정반대라 `target_names`를 안 찍고 `classification_report`를 읽으면 recall·precision을 전부 거꾸로 해석하게 된다. 관심 있는 쪽(악성=0)의 recall이 필요하면 `recall_score(..., pos_label=0)`으로 sklearn 기본값을 뒤집어야 함.

### 3) 정확도 하나로 끝내지 않고 혼동행렬로 "틀린 방향"을 본다

```
[[40  2]      악성을 양성(건강)으로 놓침 2건 (FN)
 [ 0 72]]     양성을 악성으로 오진 0건 (FP)
```
정확도 98.25%는 훌륭해 보이지만 틀린 2건이 전부 최악의 방향(FN)에 몰려 있었다. 의료 문제에서는 정확도 숫자보다 "어느 방향으로 틀렸는가"가 훨씬 중요하다는 걸 확인 — 이후 임계값을 0.9로 올려 정확도 4%p를 내주고 놓친 악성을 1명 줄이는 트레이드오프까지 실제로 계산해봄.

### 4) GridSearchCV의 "최적 점수"는 진짜 검증 점수가 아닐 수 있다

```
GridSearchCV(Ridge(), {'alpha':[...]}, cv=5).fit(X_tr, y_tr)
최적 CV R2 : 0.5267   <- train 내부 교차검증 점수
검증 R2    : 0.33     <- 한 번도 안 본 X_val 실제 점수
```
`best_score_`는 train을 다시 쪼갠 교차검증 점수이지, 완전히 새로운 데이터에 대한 점수가 아니다. 두 값의 차이가 클 수 있다는 걸 실측으로 확인했으니, "CV 점수가 왜 이렇게 높지?"라는 의심이 들 때는 파이프라인으로 전처리까지 폴드 안에 가두고 재검증하는 습관이 필요함.

### 5) 튜닝은 공짜가 아니다 — 후보 수 × fold 수만큼 비용이 든다

```
baseline (RandomForest 기본값)      : CV R2 0.3118
GridSearchCV (n_estimators×max_depth 6조합 × cv=5 = 30번 학습) : CV R2 0.3181
```
개선폭이 0.006에 불과한데 학습은 30번이나 돌렸다. 튜닝 후보를 늘릴수록 비용은 배로 늘지만 성능 향상은 데이터 품질에 달려 있는 경우가 많아, 향상폭이 작다면 하이퍼파라미터를 더 뒤지기보다 데이터·특성 쪽 개선을 먼저 검토하는 게 효율적이라는 걸 확인.
