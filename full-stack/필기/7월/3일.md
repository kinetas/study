## 05_UPLOAD — 파일 업로드 / 다운로드

### Spring Boot — `UpDownloadController.java`

```java
@CrossOrigin(origins = {"http://127.0.0.1:3000","http://localhost:3000"})
@RestController
public class UpDownloadController {

    @Value("${root.path}")   private String Root;
    @Value("${upload.path}") private String Upload;

    // 업로드: MultipartFile[] 로 여러 파일 수신
    @PostMapping(value="/upload", consumes=MediaType.MULTIPART_FORM_DATA_VALUE,
                                  produces=MediaType.APPLICATION_JSON_VALUE)
    public void upload(@RequestParam("files") MultipartFile[] files) throws IOException {
        String uploadPath = Root + File.separator + Upload + File.separator;
        File dir = new File(uploadPath);
        if (!dir.exists()) dir.mkdirs();

        for (MultipartFile file : files) {
            file.transferTo(new File(dir, file.getOriginalFilename()));
        }
    }

    // 파일 목록 조회
    @GetMapping("/upload/list")
    public ResponseEntity<List<String>> filelist() {
        File[] fileList = new File(Root + File.separator + Upload).listFiles();
        List<String> list = Arrays.stream(fileList)
                .map(el -> Root + File.separator + Upload + File.separator + el.getName())
                .collect(Collectors.toList());
        return new ResponseEntity<>(list, HttpStatus.OK);
    }

    // 다운로드: 파일 경로를 쿼리 파라미터로 받아 바이너리 스트림 반환
    @GetMapping(value="/download", produces=MediaType.APPLICATION_OCTET_STREAM_VALUE)
    public ResponseEntity<Resource> download(@RequestParam("filePath") String filePath)
            throws UnsupportedEncodingException {
        Resource resource = new FileSystemResource(filePath);
        String filename = resource.getFilename();
        HttpHeaders headers = new HttpHeaders();
        // 한글 파일명 깨짐 방지: UTF-8 → ISO-8859-1 인코딩
        headers.add("Content-Disposition",
            "attachment; filename=" + new String(filename.getBytes("UTF-8"), "ISO-8859-1"));
        return new ResponseEntity<>(resource, headers, HttpStatus.OK);
    }
}
```

### React — `Upload.jsx` + `List.jsx`

```jsx
// Upload.jsx: FormData로 여러 파일 전송
const Upload = () => {
    const [files, setFiles] = useState([]);

    const uploadHandler = () => {
        const formData = new FormData();
        for (let idx in files) formData.append("files", files[idx]);

        axios.post("http://localhost:8080/upload", formData,
            { headers: { "Content-Type": "multipart/form-data" } })
        .then(response => { console.log(response.data) })
        .catch(error => { navigate('/exception', { state: { message: error.response.data.message } }) });
    };

    return (
        <div>
            {/* multiple: 여러 파일 선택 가능 */}
            <input type="file" name="files" onChange={e => setFiles(e.target.files)} multiple />
            <button onClick={uploadHandler}>UPLOAD</button>
        </div>
    );
};

// List.jsx: 파일 목록 + 클릭 시 다운로드
const List = () => {
    const [fileList, setFileList] = useState([]);

    useEffect(() => {
        axios.get("http://localhost:8080/upload/list")
             .then(resp => setFileList(resp.data));
    }, []);

    const handleDownload = (e) => {
        const filePath = e.target.getAttribute("data-filename");

        axios.get("http://localhost:8080/download",
            {
                params: { filePath },
                responseType: "blob",              // 바이너리 응답을 Blob으로 수신
                headers: { "Content-Type": "application/json" }
            })
        .then(resp => {
            const filename = filePath.split("/").pop();
            const url  = window.URL.createObjectURL(new Blob([resp.data]));
            const link = document.createElement("a");
            link.href = url;
            link.setAttribute("download", filename);
            document.body.appendChild(link);
            link.click();
            link.parentNode.removeChild(link);    // DOM 정리
        });
    };

    return (
        <div>
            {fileList.map((el, idx) => (
                <div key={idx}>
                    <Link onClick={handleDownload} data-filename={el}>{el}</Link>
                </div>
            ))}
        </div>
    );
};
```

| 포인트 | 설명 |
|--------|------|
| `FormData` | 파일을 `multipart/form-data` 형식으로 묶는 객체 |
| `formData.append("files", file)` | Spring의 `@RequestParam("files")` 와 이름 일치 필수 |
| `responseType: "blob"` | 다운로드 시 바이너리 응답을 Blob으로 받음 |
| `createObjectURL(blob)` | Blob → 가상 URL 생성 → `<a>` 태그 클릭으로 저장 |

---

### 구조 — React는 Spring Boot만 바라본다

```
React (axios)
  ↓ GET http://localhost:8080/Open/Weather
Spring Boot (RestTemplate)
  ↓ GET https://apis.data.go.kr/...?serviceKey=...
외부 API 서버
  ↓ JSON 응답
Spring Boot (응답 가공)
  ↓ ResponseEntity<Map>
React (결과 표시)
```

> 이 구조의 이점: API Key가 React 코드(브라우저)에 노출되지 않음. React는 항상 자신의 Spring Boot 서버만 호출한다.

## 07_SPRINGSECURITY — Spring Security + JWT(쿠키) + Redis Refresh Token

---

### 인증 방식 — 쿠키에 담은 JWT

`Authorization: Bearer` 헤더 대신 **HttpOnly 쿠키**에 Access Token을 담는 방식을 택했다. 헤더+LocalStorage 방식은 XSS에 취약하기 때문.

```
로그인 성공
  → JWTTokenProvider.generateToken()  — Access Token + Refresh Token 생성
  → Refresh Token은 Redis에 저장 ("RT:" + username)
  → Access Token은 HttpOnly 쿠키(accessCookie)로 응답, username도 쿠키로 같이 내려줌
```

### `SecurityConfig.java` — 필터체인 핵심 설정

```java
http.csrf(config -> config.disable());   // logout이 기본 POST 요구하는 것도 CSRF 끄면 완화됨

http.authorizeHttpRequests(auth -> {
    auth.requestMatchers("/favicon.ico").permitAll();
    auth.requestMatchers("/","/join","/login","/validate","/logout").permitAll();
    auth.requestMatchers("/user").hasAnyRole("USER","ADMIN");
    auth.requestMatchers("/manager").hasAnyRole("MANAGER");
    auth.requestMatchers("/admin").hasAnyRole("ADMIN");
    auth.anyRequest().authenticated();
});

http.formLogin(login -> login.disable());   // 로그인은 UserRestController에서 직접 처리(직접 인증)

http.logout(logout -> {
    logout.permitAll();
    logout.addLogoutHandler(customLogoutHandler);          // 쿠키/토큰 정리
    logout.logoutSuccessHandler(customLogoutSuccessHandler); // 응답 마무리(redirect)
});

http.exceptionHandling(exception -> {
    exception.authenticationEntryPoint(customAuthenticationEntryPoint); // 401
    exception.accessDeniedHandler(customAccessDeniedHandler);           // 403
});

http.oauth2Login(oauth2 -> oauth2.loginPage("/login"));

http.sessionManagement(s -> s.sessionCreationPolicy(SessionCreationPolicy.STATELESS));

http.addFilterBefore(jwtAuthorizationFilter, LogoutFilter.class); // 매 요청 JWT 검사
```

> `formLogin`을 끄고 `UserRestController`에 `/login`을 직접 만든 이유: 로그인 성공 시 JWT 발급 + 쿠키 세팅 같은 커스텀 로직이 필요해서, Security 기본 `UsernamePasswordAuthenticationFilter` 흐름 대신 컨트롤러에서 `authenticationManager.authenticate()`를 직접 호출한다.

### `UserRestController.java` — 실제 엔드포인트

| 엔드포인트 | 설명 |
|---|---|
| `POST /join` | 회원가입, 비밀번호 BCrypt 인코딩 |
| `POST /login` | `authenticationManager.authenticate()` → JWT 발급 → Redis에 Refresh 저장 → Access/username 쿠키 세팅 |
| `GET /user` | 로그인된 사용자 정보 조회 (role 기반 접근 제한) |
| `POST /logout` | **본문은 비어있는 placeholder.** 실제 처리는 `LogoutFilter`가 먼저 가로챈다 (아래 CORS 이슈 참고) |
| `GET /validate` | 프론트가 페이지 로드 시 "지금 로그인 상태인지" 확인하는 용도. `permitAll` + 컨트롤러 내부에서 `authentication.isAuthenticated()` 직접 체크 |

### `JWTAuthorizationFilter.java` — 요청마다 도는 인증 필터

```
쿠키에서 accessToken, username 추출
  ├─ accessToken 없음 → (username 있으면) Redis에서 refreshToken 조회 → 유효하면 accessToken 재발급 + 쿠키 갱신
  ├─ accessToken 유효 → SecurityContextHolder 에 Authentication 세팅
  └─ accessToken 만료(ExpiredJwtException) → Redis에서 refreshToken 조회
        ├─ refreshToken 유효 → accessToken 재발급 + 쿠키 갱신
        └─ refreshToken도 만료 → 쿠키 삭제 + Redis에서 삭제 (재로그인 필요)
```

### 오늘 겪은 CORS 트러블슈팅 (핵심)

`@CrossOrigin`은 **요청이 실제로 컨트롤러까지 도달했을 때만** 적용된다. Spring Security 필터체인 단계에서 응답이 먼저 끊기면 아무 소용이 없다는 걸 세 가지 실제 사례로 확인했다.

**1) `Access-Control-Allow-Credentials`가 빈 문자열로 옴 (`/login`)**

- 원인: `@CrossOrigin(origins = "*")` — 와일드카드 origin은 브라우저 스펙상 credentials(쿠키) 허용 헤더를 함께 못 보낸다.
- 해결: `@CrossOrigin(origins = "http://localhost:3000", allowCredentials = "true")` 로 origin을 명시.

**2) `/validate`는 CORS 헤더가 아예 없음**

- 원인: `/validate`가 `anyRequest().authenticated()`에 걸려서, 인증 안 된 상태로 요청하면 `CustomAuthenticationEntryPoint.commence()`가 `response.sendRedirect(...)`로 바로 응답을 끊어버린다. 이 시점은 `DispatcherServlet`(컨트롤러) 도달 **이전**이라 `@CrossOrigin`이 적용될 기회 자체가 없다.
- 해결: `/validate`를 `permitAll`에 추가 — 인증 여부와 무관하게 컨트롤러까지 도달시키고, 컨트롤러 내부의 `validateToken()`이 직접 인증 여부를 검사해서 200/401을 반환하도록 함(이미 그렇게 구현되어 있었음).

**3) `/logout`은 프리플라이트(OPTIONS) 자체가 302로 리다이렉트됨**

- 원인: `/logout`은 컨트롤러가 아니라 Spring Security의 `LogoutFilter`가 직접 처리하는데, 이 필터는 기본적으로 `POST`만 매칭한다. 그래서 `OPTIONS /logout`(프리플라이트)은 `LogoutFilter`를 그냥 통과해버리고 `anyRequest().authenticated()`에 걸려 `CustomAuthenticationEntryPoint`로 리다이렉트되어 CORS 헤더 없는 302가 응답됨. 브라우저 Network 탭에서 `OPTIONS /logout → 302 Found → location: /login?error=...`로 직접 확인.
- 해결(2단):
  - `/logout`을 `permitAll`에 추가 → 프리플라이트가 인증 체크에 걸리지 않게 함.
  - `UserRestController`에 실행되지 않는 `@PostMapping("/logout")` placeholder 메서드를 추가 → `RequestMappingHandlerMapping`이 `/logout` 경로를 인식하게 되어, `@CrossOrigin`이 프리플라이트 응답에도 적용됨. 실제 POST 요청은 여전히 `LogoutFilter`가 먼저 가로채서 placeholder 메서드는 호출되지 않는다.
  - 추가로 `CustomLogoutSuccessHandler`에서도 `Origin` 헤더를 그대로 반사해 `Access-Control-Allow-Origin` / `Access-Control-Allow-Credentials`를 직접 세팅(방어적 조치).

**교훈 정리**

| 상황 | `@CrossOrigin` 적용 여부 |
|---|---|
| `permitAll` 경로 → 컨트롤러까지 정상 도달 | ✅ 적용됨 |
| 인증 필요 경로, 인증 실패 → EntryPoint가 즉시 응답(redirect 등) | ❌ 컨트롤러 도달 전이라 적용 안 됨 |
| Security의 내장 필터(`LogoutFilter` 등)가 직접 처리하는 경로 | ❌ 애초에 컨트롤러가 없어서 적용 안 됨 |

> 더 "정석"에 가까운 대안은 `SecurityConfig`에 `http.cors(cors -> cors.configurationSource(...))`를 붙여 Security 필터체인 자체에 CORS를 연동하는 것(필터 초반에 CORS 헤더가 붙어서 인증 성공/실패와 무관하게 항상 적용됨). 이번 프로젝트에서는 `@CrossOrigin` 어노테이션 방식을 유지하기로 하고, 위처럼 permitAll 확장 + placeholder 매핑 + 핸들러 내 수동 헤더 설정으로 우회했다.

## 3일 핵심 정리

---

### 1) 파일 업로드 / 다운로드 핵심

```
업로드  FormData.append("files", file)
        axios.post(url, formData, { headers: { Content-Type: multipart/form-data } })
        → @RequestParam("files") MultipartFile[]

다운로드  axios.get(url, { params: { filePath }, responseType: "blob" })
          → new Blob([resp.data]) → createObjectURL → <a> 클릭
```

---

### 2) 외부 API 연동 — Spring Boot 프록시 패턴

```
React → Spring Boot → 외부 API

이유:
  - API Key가 브라우저에 노출되지 않음
  - CORS 문제 없음 (서버 간 통신은 CORS 무관)
  - 응답 가공 후 React에 필요한 데이터만 전달

RestTemplate 순서:
  URI(UriComponentsBuilder) → HttpHeaders → HttpEntity → exchange() → getBody()
```

---

### 3) Spring Security + JWT 인증 흐름

```
[로그인]  React → POST /login → authenticationManager.authenticate()
          → JWT(Access+Refresh) 발급 → Refresh는 Redis, Access는 HttpOnly 쿠키

[인증 유지]  매 요청 → JWTAuthorizationFilter → 쿠키의 accessToken 검증
             → 만료 시 Redis의 refreshToken으로 재발급 → SecurityContextHolder 세팅

[상태 확인]  React 진입 시 → GET /validate (permitAll) → 컨트롤러가 인증 여부 직접 판단

[로그아웃]  React → POST /logout → LogoutFilter가 가로챔
            → CustomLogoutHandler / CustomLogoutSuccessHandler
            → Redis refresh 삭제 + 쿠키 만료 + redirect
```

---

### 4) CORS 디버깅 체크리스트 (Spring Security 조합 시)

```
증상별 원인 추적:
  1. "Access-Control-Allow-Credentials가 빈 문자열"
     → @CrossOrigin(origins="*") + withCredentials:true 동시 사용 여부 확인

  2. "CORS 헤더 자체가 없음" (401/403/302 응답)
     → 그 경로가 permitAll인지 확인
     → 아니라면: 인증 실패 시 EntryPoint/AccessDeniedHandler가 컨트롤러 도달 전에
       응답을 끊는 게 원인일 가능성 높음

  3. "특정 시스템 경로(logout 등)에서 프리플라이트부터 실패"
     → 그 경로가 컨트롤러가 아니라 Security의 내장 필터(LogoutFilter 등)가
       처리하는 경로인지 확인 → 필터의 매처가 method를 제한하는지 확인
       (예: POST만 매칭 → OPTIONS는 필터를 통과해 인증 체크로 흘러감)

  4. 브라우저 개발자도구 Network 탭에서 실제 OPTIONS 요청의
     Status Code / Response Headers를 직접 확인하는 게 제일 빠름
     (추측으로 코드만 고치지 말 것)
```
