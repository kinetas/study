# 4월 9일 필기 — 데이터베이스 (database.md)

## 핵심요약

INSERT는 여러 행을 한 번에 삽입하거나 INSERT IGNORE·ON DUPLICATE KEY UPDATE로 PK 중복 상황을 처리하며, AUTO_INCREMENT로 자동 번호를 부여한다. 제약조건으로 PK(단일·복합), FK(CASCADE·RESTRICT·SET NULL 옵션)를 생성·삭제하며, 외래키 순서 문제는 FOREIGN_KEY_CHECKS를 이용해 처리할 수 있다(운영에서는 주의 필요). 변수·형변환(CAST)·내장함수(문자열: CONCAT·SUBSTRING·TRIM, 날짜: NOW·YEAR·MONTH)로 데이터를 가공하고, LOAD_FILE·INTO DUMPFILE로 BLOB 파일을 DB에 넣고 내보낼 수 있다. 인덱스는 조건 검색 성능을 높이는 구조로 PK에는 클러스터형, 그 외에는 보조 인덱스가 생성되며 쓰기·저장 비용 증가를 감안해 설계해야 한다.


예제 SQL 전체는 `SQL_use/01select.sql`, `SQL_use/02select.sql` 참고. **INSERT·UPDATE·DELETE·제약**은 `03insert-update-delete.sql`, `04constraint.sql`, **변수·형변환**은 `05type.sql`, **내장 함수**는 `06func.sql`, **BLOB·파일**은 `07updown.sql`, **인덱스**는 `08index.sql` 및 아래 요약 참고. 아래 SELECT 요약은 **shopdb**(`usertbl`, `buytbl`) 및 일부 **world** DB 기준 개념 묶음.
**다른 DB(`world`)**: `city`에서 `countrycode`별 `population` 합계, `country`에서 `continent`별 `lifeexpectancy` 평균 등 **같은 패턴**의 `GROUP BY` 적용 예.

## INSERT·UPDATE·DELETE 실습 요약 (`03insert-update-delete.sql`)

예제 SQL 전체는 `SQL_use/03insert-update-delete.sql` 참고. (스크립트 일부는 주석 처리 상태이므로, 실행 시 주석 해제 후 순서대로 시도.)

### `DELETE`

- `DELETE FROM 테이블명;` — `WHERE` 없이 실행하면 **전체 행**이 대상이 될 수 있어 운영 DB에서는 특히 주의 필요.

### `INSERT` 패턴 (MySQL)

| 패턴 | 설명 |
|------|------|
| 여러 행 한 번에 | `INSERT INTO 테이블 VALUES (…), (…), (…);` |
| PK(또는 유니크) 중복 시 **행 무시** | `INSERT IGNORE INTO …` — 충돌 나는 행은 삽입하지 않고 넘어감 |
| 자동 번호 | `AUTO_INCREMENT` 가 걸린 컬럼에 `NULL`(또는 생략 규칙에 맞게) 넣어 연속 번호 부여 |
| 중복 시 **갱신**(업서트) | `INSERT INTO … VALUES … ON DUPLICATE KEY UPDATE 컬럼 = 표현식;` — PK/유니크 충돌 시 `UPDATE` 절 실행 (예: `amount = amount + 1`) |

### `AUTO_INCREMENT` 확인·조정

| 목적 | 예시 개념 |
|------|-----------|
| 다음 증가값 조회 | `information_schema.tables` 에서 `AUTO_INCREMENT` (스키마·테이블명 조건) |
| 방금 넣은 자동값 | `SELECT LAST_INSERT_ID();` |
| 기준값 변경 | `ALTER TABLE … AUTO_INCREMENT = n;` (데이터·스토리지 상태에 따라 기대와 다를 수 있어 문서·환경 확인) |

## 제약 조건 실습 요약 (`04constraint.sql`)

예제 SQL 전체는 `SQL_use/04constraint.sql` 참고.

### 기본키(PK)

| 방법 | 예시 개념 |
|------|-----------|
| 컬럼 정의에 직접 | `id … PRIMARY KEY` |
| 테이블 제약으로 단일·복합 | `PRIMARY KEY (id)` / `PRIMARY KEY (id, name)` |
| 나중에 추가 | `ALTER TABLE … ADD CONSTRAINT 이름 PRIMARY KEY (컬럼…);` |
| 제거 | `ALTER TABLE … DROP PRIMARY KEY;` |

- 복사본 테이블(`CREATE TABLE … SELECT …`)에 데이터를 넣은 뒤 `num` 등에 **이름 붙인 PK**(`PK_num` 등)를 거는 패턴 정리.
- 메타 확인: `DESC`, `SHOW CREATE TABLE`, `information_schema.columns` 의 `COLUMN_KEY` 등.

### 외래키(FK)

| 방법 | 예시 개념 |
|------|-----------|
| 생성 시 | `CONSTRAINT FK_이름 FOREIGN KEY (자식컬럼) REFERENCES 부모테이블(부모컬럼)` |
| 복합 FK | `FOREIGN KEY (id, name) REFERENCES 부모(id, name)` |
| 나중에 추가 | `ALTER TABLE … ADD CONSTRAINT … FOREIGN KEY … REFERENCES …` |

**참조 시 `ON UPDATE` / `ON DELETE` 옵션**(개념 정리 — 제품·버전별 세부는 매뉴얼 확인):

| 옵션 | 요지 |
|------|------|
| `CASCADE` | 부모 쪽 PK 값 변경·행 삭제가 자식 FK에 **연쇄 반영** |
| `RESTRICT` / `NO ACTION` | 참조가 있으면 부모 삭제·갱신을 **막는** 쪽(엔진에서 동작 차이가 있을 수 있음) |
| `SET NULL` | 부모 변경 시 자식 FK를 **NULL** 로 |
| `SET DEFAULT` | 지정된 **기본값**으로 |

- 정의 확인: `information_schema.referential_constraints`, `SHOW CREATE TABLE` 등.

### FK 제거·인덱스(MySQL)

- `ALTER TABLE … DROP FOREIGN KEY 제약이름;`
- FK에는 보통 **인덱스**가 함께 생성됨. 필요 시 `DROP INDEX` 로 처리. (예시에서는 FK와 연관된 인덱스명 처리)
- `SHOW INDEX FROM 테이블;`

### 테이블 삭제 순서·외래키 검사 끄기

- **자식(FK가 있는 쪽) 테이블을 먼저** `DROP` 하고, 참조되는 **부모**를 나중에 삭제하는 일반적 순서.
- 개발·데이터 정리 시에만: `SET FOREIGN_KEY_CHECKS = 0;` … `DROP TABLE …;` … `SET FOREIGN_KEY_CHECKS = 1;` 로 순서를 바꿀 수 있으나, **운영에서는 오용 시 무결성 훼손 위험**으로 매우 높은 주의 필요.

### 예시: `copy_buytbl`

- `buytbl` 복사 후 `num`에 PK, `userID` → `usertbl(userID)` FK 를 두고, `ON UPDATE CASCADE`, `ON DELETE RESTRICT` 등으로 규칙을 맞추는 예.

## 변수·형변환 (`05type.sql`)

예제 SQL 전체는 `SQL_use/05type.sql` 참고.

### 사용자 변수(MySQL)

- `SET @이름 = 값;` — 세션 동안 쓰는 **사용자 변수** (`@var1` 등).
- `SELECT` 절에서 `@변수`와 컬럼을 함께 쓰거나, `@var1 + @var2` 처럼 연산하는 방식.

### 준비된 문(`PREPARE` / `EXECUTE`)

- 문자열로 SQL을 만들고 `PREPARE 이름 FROM '…'` → `EXECUTE 이름 USING @변수` 로 플레이스홀더(`?`)에 값을 넣는 패턴 정리. (예: `ORDER BY … LIMIT ?`)

### 형변환

| 구분 | 설명 |
|------|------|
| 암시적(자동) | 연산 시 DB가 타입을 맞춤(데이터 손실을 줄이는 쪽으로 동작하는 경우가 많음) |
| 명시적(강제) | `CAST(값 AS 타입)` 등으로 개발자가 지정(목적에 따라 손실 가능) |

- 날짜 문자열: `CAST('2024$01$01' AS DATE)` 처럼 구분 문자가 달라도 날짜로 해석되는 경우 정리. (환경·형식에 따름)
- `CONCAT` 전에 `CAST(price AS CHAR(10))` 등으로 숫자를 문자열로 맞추는 예 정리.

### 숫자·문자 연산(참고)

- `'100' + 200` 처럼 **숫자로 읽히는 앞부분**만 더해지는 등 MySQL의 변환 규칙 적용 예.
- 비교 결과를 **참 1 / 거짓 0** 으로 쓰는 형태 정리.

## 내장 함수 (`06func.sql`)

예제 SQL 전체는 `SQL_use/06func.sql` 참고.

### 문자열

| 함수 | 용도 |
|------|------|
| `CONCAT(…)` | 여러 값을 이어 붙임 |
| `CONCAT_WS(구분자, …)` | 구분자로 연결 |
| `SUBSTRING(str, 시작)` / `SUBSTRING(str, 시작, 길이)` | 부분 문자열 |
| `SUBSTRING_INDEX(str, 구분자, n)` | 구분자 기준 n번째까지(예: 날짜에서 `'-'` 기준 앞부분) |
| `LENGTH(…)` | 바이트 길이(문자셋에 따라 문자 “개수”와 다를 수 있음) |
| `TRIM(…)` | 앞뒤 공백 제거 |
| `LOWER` / `UPPER` | 대소문자 변환 |

- 예: `bin(숫자)` — 이진 표기 확인 등 교육·디버그용 예.

### 날짜·시간

| 함수 | 용도 |
|------|------|
| `YEAR` / `MONTH` / `DAY` | 날짜 컬럼에서 연·월·일 추출 |
| `NOW()` | 현재 일시 |
| `CURDATE()`, `CURTIME()` | 오늘 날짜, 현재 시각 |
| `DATE(…)`, `TIME(…)` | 일시에서 날짜·시각만 추출 |

- `REPLACE` 로 날짜·시간 구분 문자를 바꾼 뒤 `CONCAT` 으로 원하는 포맷(예: `YYYY#MM#DD` 와 `hh|mm|ss` 결합)을 만드는 예 정리.

## BLOB·파일 입출력 (`07updown.sql`)

예제 SQL 전체는 `SQL_use/07updown.sql` 참고. (`testdb` 등 별도 DB·경로는 환경에 맞게 수정)

### 테이블 예

- `LONG BLOB` 컬럼에 **파일 바이너리**를 넣는 형태.

### 서버에서 파일 읽기 → DB에 넣기

- `INSERT INTO … VALUES ('제목', LOAD_FILE('경로'));` — 서버가 접근 가능한 경로의 파일을 읽어 저장하는 방식.
- Windows 경로는 `c:\\sql\\파일명` 처럼 **이스케이프**하거나 슬래시를 쓰는 식으로 맞추는 방식.
- `LOAD_FILE` 은 **권한·secure_file_priv·경로** 제약이 있어 실패 시 서버 설정 확인 필요.

### DB에서 파일로 내보내기

- `SELECT blob컬럼 FROM … WHERE … INTO DUMPFILE '경로';` — **단일 행** 결과를 바이너리로 파일에 쓰는 방식.
- `INTO OUTFILE` 과 다른 용도. (텍스트 내보내기 등) 보안·권한·경로 정책 확인 필요.

## 인덱스 (`08index.sql`)

예제 SQL 전체는 `SQL_use/08index.sql` 참고.

### 목적

- **`WHERE`** 등 조건에 자주 쓰는 열에 인덱스를 둘 때 **검색·조인 비용** 감소 가능성이 큰 편. (항상 빨라지는 것은 아니며, 쓰기·저장 비용 증가는 가능)

### MySQL에서 자주 언급되는 유형(개념)

| 유형 | 요지 |
|------|------|
| B-Tree | InnoDB 등에서 **기본**에 가깝게 쓰이는 범용 인덱스 |
| Hash | **동등 비교**에 유리한 경우(범위·LIKE 등은 부적합할 수 있음) |
| Full-text | 텍스트 **전문 검색** |
| Spatial | 공간(위치) 데이터 |

### 클러스터형 vs 보조(보통 InnoDB 맥락)

- **PK(클러스터형 인덱스)**: 테이블당 **하나**, 데이터 정렬·저장과 연관되는 방식으로 이해.
- **보조 인덱스**: PK 외 컬럼·유니크·수동 생성 인덱스 등, **여러 개** 둘 수 있는 구조.

### 제약과 인덱스

- **PK**, **`UNIQUE`** 를 걸면 보통 **유니크 인덱스**가 함께 생성됨. `SHOW INDEX FROM 테이블;` 로 확인.

### 생성·삭제

| 작업 | 예시 개념 |
|------|-----------|
| 유니크(단일·복합) | `UNIQUE INDEX 이름 (col)` / `UNIQUE INDEX 이름 (col2, col3)` |
| 일반 보조 | `CREATE INDEX 이름 ON 테이블(컬럼);` |
| PK 제거 | `ALTER TABLE … DROP PRIMARY KEY;` |
| 인덱스 제거 | `ALTER TABLE … DROP INDEX 인덱스명;` (유니크 제약 이름과 인덱스명이 같은 경우가 많음) |

> MySQL 버전에 따라 `DROP CONSTRAINT` 로 유니크를 제거하는 문법 차이 가능성. **해당 버전 매뉴얼**과 `SHOW CREATE TABLE` 결과 병행 확인.

### FK와 인덱스

- FK를 걸면 **참조 열에 인덱스가 생기는** 경우가 많음. `SHOW INDEX` 로 함께 확인.

### 성능 확인 예(`employees`)

- 큰 테이블(`salaries` 등)에서 특정 날짜(`to_date`)로 조회할 때 **인덱스 생성 전후** 비교 예. 실제 효과는 데이터 분포·쿼리 패턴에 따라 차이.
